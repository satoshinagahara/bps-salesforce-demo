# 商談類似度分析 LWCパネル設計

**目的**：商談レコードページに配置するLWCパネル。Agentforceチャットの対話的な深掘りとは別に、構造化されたUIで類似商談の分析結果を表示する。
**作成日**：2026-03-25
**関連**：`docs/in-progress/opportunity-similarity-implementation.md`

---

## 1. 設計思想

### Agentforce vs LWCパネルの役割分担

| | Agentforce チャット | LWCパネル |
|---|---|---|
| 入力 | 自然言語（チャット） | 自然言語（テキストボックス） |
| 出力 | マークダウンテキスト | 構造化カード＋アクションボタン |
| 用途 | 対話的な深掘り・フォローアップ質問 | 一覧性の高い分析結果表示＋アクション実行 |
| 強み | 会話の文脈を保持、連続質問 | 視認性、リンク、ToDo作成などのUI操作 |

両方から同じApex（`OpportunitySimilaritySearch`）を呼び出す。バックエンドは共通。

### 中心的な考え方

「類似商談を並べる」のではなく、**ユーザーの判断と行動を支援するカード**を提示する。

- 各カードは「推奨アクション」単位
- 根拠となる類似商談はカード内にリンクとして埋め込む
- 具体的なアクション項目はToDoとして作成可能

### 2.2 入力状況セクション

パネル初期表示時に、現在の商談のデータ入力状況を表示する。Prompt Template呼び出し不要（SOQLのCOUNTのみ）。

**目的：**
- ユーザーに「この分析の信頼度」を事前に伝える
- データが不足している場合、入力を促す（Sparseデータ問題への直接的な解決策）
- 「なぜ示唆の精度が低いのか」をユーザーが理解できる

**表示要素：**

| 項目 | 取得方法 | 表示 |
|---|---|---|
| サマリカード | Opportunity_Summary__c の有無 | ✅ あり / ❌ なし |
| Description | Opportunity.Description の有無 | ✅ あり / ❌ なし |
| 商品登録 | OpportunityLineItem の COUNT | ✅ N件 / ❌ 0件 |
| 活動記録 | Task + Event の COUNT | ✅ N件 / ⚠️ N件（少ない）/ ❌ 0件 |

**信頼度スコア（5段階）：**

| レベル | 条件 | 表示 |
|---|---|---|
| 5 (高い) | サマリカード＋Description＋商品3件以上＋活動5件以上 | ●●●●● |
| 4 | サマリカード＋Description＋商品or活動あり | ●●●●○ |
| 3 | Descriptionあり＋商品or活動あり | ●●●○○ |
| 2 | Descriptionのみ | ●●○○○ |
| 1 (低い) | Descriptionなし | ●○○○○ |

**信頼度が低い場合のメッセージ例：**
- レベル1-2: 「Descriptionと活動を充実させると、より精度の高い示唆が得られます」
- サマリカードなし: 「サマリカードが未生成です。分析精度を上げるには管理者にご連絡ください」

## 3. データフロー

### 3.1 Agentforce向け（既存・変更なし）

```
ユーザー質問 → Agent LLM → Apex Action
  → SOQL検索 + ConnectApi(Prompt Template) → マークダウンテキスト
  → Agentforceチャットに表示
```

### 3.2 LWCパネル向け（新規）

```
ユーザー質問 → LWC → Apex Controller（@AuraEnabled）
  → SOQL検索 + ConnectApi(Prompt Template) → 構造化JSON
  → LWCがカードとしてレンダリング
```

### 3.3 共通化のポイント

- SOQL検索ロジック（`searchSimilar`メソッド）は共通
- Prompt Templateは2つ用意：
  - `OpportunitySimilarityInsights`（既存）: マークダウンテキスト出力 → Agentforce用
  - `OpportunitySimilarityStructured`（新規）: JSON出力 → LWC用
- または1つのPrompt TemplateでJSON出力し、Agentforce側はAgent LLMがテキスト化

### 3.4 推奨: 1 Prompt Template + 出力フォーマット指定

Prompt Templateは1つにして、JSON形式で出力させる。

- **LWC**: JSONをそのままパースしてカード表示
- **Agentforce**: Apex内でJSONからマークダウンに変換（or Agent LLMに任せる）

これにより分析ロジックの一貫性を保てる。

## 5. ToDo作成機能

### 動作

1. ユーザーが推奨アクションカードの「ToDoとして作成」ボタンをクリック
2. LWCがApex経由でTaskレコードを作成
   - `Subject`: アクション項目のテキスト
   - `WhatId`: 現在の商談ID
   - `Status`: "Not Started"
   - `ActivityDate`: 1週間後（デフォルト）
3. 作成完了のトーストを表示

### 考慮事項

- 1カードに複数のアクション項目がある場合、まとめて1つのToDoにするか、個別に作るかはUI上で選択可能にする
- OwnerId はログインユーザー（デフォルト）

### Apexコントローラー

```
OpportunitySimilarityController.cls    ← @AuraEnabled メソッド
  - getInputStatus(opportunityId) → InputStatus JSON（パネル初期表示時）
  - searchAndAnalyze(opportunityId, userQuery) → Analysis JSON（分析実行時）
  - createTodoFromRecommendation(opportunityId, subject) → Task Id
```

`getInputStatus` は軽量（SOQLのCOUNTのみ）で、パネルが表示された瞬間に呼ばれる。
`searchAndAnalyze` はPrompt Template呼び出しを含むため時間がかかる。ユーザーが「分析」ボタンを押した時のみ実行。

## 8. テストで発見された設計課題（2026-03-25）

### 問題: 信頼度の低い商談が参考事例として提示される

類似商談リンクを辿った先が、現在の商談よりフェーズが手前で活動も乏しい商談だった。

**根本原因：**
- 類似商談の検索が構造化フィールド（deal_type, sales_motion等）のみでマッチングしており、**参照先商談のデータ品質を考慮していない**
- 14件のテストデータのうちSparseなもの（A-4, D-2等）も候補に含まれる
- フェーズの進捗度も考慮されていない

**設計変更方針：**

サマリカード（Opportunity_Summary__c）に信頼度関連フィールドを追加し、以下を実現する：

1. **検索時フィルタ**: `Confidence_Score__c >= 3` で低品質商談を除外
2. **フェーズフィルタ**: 現在の商談と同等以上のフェーズの商談のみ返す（Closed Wonは最も参考価値が高い）
3. **入力状況表示の統一**: LWCの入力状況セクションもサマリカードの信頼度を参照
4. **類似商談カードに信頼度表示**: 参照先の品質をユーザーが事前判断可能に

**追加フィールド（Opportunity_Summary__c）：**

| フィールド | 型 | 内容 |
|---|---|---|
| `Confidence_Score__c` | Number | 信頼度スコア（1-5） |
| `Product_Count__c` | Number | 商品登録件数（生成時スナップショット） |
| `Activity_Count__c` | Number | 活動件数（生成時スナップショット） |
| `Has_Description__c` | Checkbox | Description有無 |
| `Opportunity_Stage__c` | Text | 生成時のフェーズ |
