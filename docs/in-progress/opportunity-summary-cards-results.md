## 2. クラスタ分離の検証

### 2.1 deal_type によるクラスタ分離

| deal_type | 該当商談 | 期待クラスタ | 一致 |

| 設備導入 | A-1, A-2, A-3, A-4, **C-2** | Cluster A | △（C-2が混入） |
| ソフトウェア/クラウド | B-1, B-2, B-3, B-4 | Cluster B | ✅ 完全一致 |
| Design Win/OEM | C-1, C-3 | Cluster C | △（C-2が抜け） |
| 保守/サービス | D-1, D-2, D-3 | Cluster D | ✅ 完全一致 |

**C-2（東南アジアメガソーラー）の分類問題**：
- Design Winというより大型調達案件。LLMは「設備導入」と判定
- 設計時にCluster Cに入れたのは「Design Win的要素あり」という意図だったが、中間表現の構造化フィールドでは正しく「設備導入」と分類された
- → **deal_type単独ではなく、複数フィールドの組み合わせでクラスタリングすべき**

### 2.2 sales_motion による営業アプローチの分類

| sales_motion | 商談 | 特徴 |
|---|---|---|
| 既存深耕 | A-1, A-2, D-1, D-2 | 既存顧客の追加案件。関係構築型が多い |
| 競合リプレース | A-3, D-3 | 既存ベンダーへの不満が動機。TCO比較が有効 |
| 新規開拓 | A-4, B-1, B-2, B-3, C-1, C-2 | 多様。PoC型/技術型/不明が混在 |

→ **sales_motionはクラスタを横断する軸**として機能。A-3（設備導入・競合リプレース）とD-3（保守・競合リプレース）は deal_type は異なるが sales_motion が同じで、営業アプローチに共通点がある。

| urgency_driver | 商談 | 示唆 |
|---|---|---|
| 事業拡大 | A-1, A-2, A-3, A-4, C-1, C-2, C-3 | 前向きな投資。ROI訴求が有効 |

## 3. 自由記述の類似度パターン（定性評価）

### 3.1 高類似度（同一クラスタ内・同一営業アプローチ）

**A-1 ≈ A-2**: 既存顧客の新拠点向け設備横展開。両者とも「既存実績→随意契約的」のパターン。

**B-1 ≈ B-4**: エネルギー管理PFの新規導入。PoC→段階展開のアプローチ、補助金/規制が動機、FAX/Excelからの脱却という課題が共通。

**C-1 ≈ C-3**: Design Win。品質認定試験がクリティカルパス、サンプル提供→評価→量産のプロセスが同じ。

**D-1 ≈ D-3**: 保守案件。ただし D-1は「データに基づく増設提案」、D-3は「SLA比較による乗り換え」とアプローチが異なる。

### 3.2 中程度の類似度（クラスタ間の関連）

**A-3 ≈ D-3**: 同一顧客（中部電設）・異なる案件性質だが、「既存ベンダー（オメガ）への不満からの乗り換え」が共通テーマ。セット提案の関係。

**B-1 ≈ B-3**: 「ソフトウェア/クラウドで設備を一元管理」が共通。ただし urgency_driver が「規制/法令」vs「事故/障害対応」で異なり、訴求ポイントが変わる。

**A-3 ≈ C-2**: 「競合あり・技術主導型・新規開拓」が共通。deal_typeは異なる（設備 vs 設備）が、営業プロセスの進め方に類似点がある。

### 3.3 低類似度（明確に異なる）

**A-1 ≠ C-1**: 設備フルパッケージ vs Design Win/OEM。商品もプロセスも決裁構造も全く異なる。

**B-4 ≠ D-1**: ソフトウェア vs 保守。顧客セグメントも動機も営業アプローチも異なる。

### 3.4 Sparseデータの類似度

**A-4**: deal_type=設備導入でCluster Aに属するが、自由記述がすべてnullのため意味的な類似度測定は不可能。「設備導入 × IT/テクノロジー × 新規開拓」という構造化フィールドのみで、B-4やC-1と間違って近い距離に出る可能性あり。

**D-2**: deal_type=保守/サービスでCluster Dに属するが、同様にnullだらけ。Closed Wonとして「参考実績」にはなるが、示唆の提供はできない。

→ **Sparseデータは「示唆を受ける側」としてのみ機能し、「示唆を提供する側」にはならない。**

## 4. 主要な発見と設計への示唆

### 発見1: 2軸の類似度が必要

- **「何を売るか」の類似度**: deal_type + product_categories で判定 → 構造化フィールドで対応
- **「どう売るか」の類似度**: sales_motion + sales_process_pattern + customer_challenge + deal_narrative で判定 → 構造化＋ベクトルのハイブリッド

営業にとって本当に参考になるのは「似たものを売った」ではなく「似た売り方をした」事例。

### 発見2: クラスタ横断の類似パターンが存在する

A-3（設備導入/競合リプレース）とD-3（保守/競合リプレース）は deal_type は異なるが「既存ベンダーからの乗り換え」という営業テーマが共通。deal_type だけでフィルタリングすると、この有用な類似性を見逃す。

### 発見3: urgency_driver が営業トークの切り口を決める

同じ Cluster B でも「規制対応」と「事故対応」では営業の切り口が全く異なる。urgency_driver は類似度の重要な軸。

### 発見4: engagement_depth とノイズの関係

ゴルフ・会食等の「ノイズ」は、engagement_depth の判定指標として活用できた。「深い」関係性が構築できている商談は、非公式接点の記録が多い傾向がある。

### 発見5: Sparse問題は「入力促進」で解決すべき

中間表現の品質は入力データの品質に直結。サマリカードの生成時に「情報不足」を明示的にフィードバックし、営業に入力を促す仕組みが必要。
