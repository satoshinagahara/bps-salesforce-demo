# アカウントプラン策定支援 × 経営ブリーフィング 設計コンセプト

**目的**：外部データを活用したアカウントプラン策定支援（現場）と、その蓄積データを横串で集約する経営ブリーフィング（経営層）を2層で構築する
**作成日**：2026-03-28
**ステータス**：構想段階（壁打ち完了、設計着手前）

---

## 1. 背景と課題

### 1.1 経営層への訴求の難しさ

Salesforceのデモは現場向けのオペレーション統合・進化を見せるものだが、「それが経営にとって何を意味するのか」を示すのが最も難しい。

従来のアプローチは「経営ダッシュボード」（Tableau等）だが、以下の構造的な問題がある：

| 問題 | 内容 |
|---|---|
| データ統合の壁 | 部門ごとにデータ定義が違う、欠損だらけ |
| 解釈の壁 | 数字とグラフを並べても「で、何をすべきか」が分からない |
| 運用の壁 | 作ったけど見ない。経営会議では別のExcel/PPTが使われる |
| 品質の壁 | データ品質を事前にETLで完璧にすることは現実的でない |

### 1.2 前提の転換

```
従来: 「経営が見るべきデータを可視化する」（ダッシュボード）
       → 集約の壁、品質の壁、解釈の壁で頓挫

転換: 「経営の問いに対して、現場データから回答を生成する」
       → データは現場で日々メンテされている（オペレーションで使っているから）
       → 集約はオンデマンドでいい（事前にETLで完璧にする必要がない）
       → LLMが「解釈」と「示唆」を担う
```

### 1.3 一貫した設計原則：「仕事の副産物としてデータが溜まる」

ニーズカードや提案書統合と同じ原則。現場の担当者に「経営のためにデータを入力しろ」と言うのではなく、**担当者が本来やるべき仕事（アカウントプラン策定）をAIが支援する副産物として、経営に必要なデータが蓄積される**。

```
ニーズカード:     営業が面談する → 副産物として市場ニーズが構造化される
提案書統合:       営業が提案書を作る → 副産物としてナレッジが抽出される
アカウントプラン:  営業が顧客戦略を練る → 副産物として戦略アライメントが可視化される
                                          ↓
                              経営ブリーフィングで横串集約
```

### 1.4 2層アーキテクチャ

```
Phase 1: アカウントプラン策定支援（現場 — Accountレコードページ）
  - 外部データ自動取得 → 業界コンテキスト
  - 顧客戦略プロファイルの作成支援
  - 戦略アライメント分析
  - アクションプラン生成
  - アカウントプランPPTのレビュー（既存Lambda + Claude Vision活用）
  ※ Phase 1だけでもアカウント担当者向けの強力なデモとして成立

Phase 2: 経営ブリーフィング（経営層 — ホームページ）
  - Phase 1で各アカウントに蓄積されたデータを横串で集約
  - 守り（供給/品質/需要）+ 攻め（市場機会）の俯瞰
  - 意思決定 → 起票 → フォロースルー
  ※ Phase 1のデータがないとPhase 2の「攻め」セクションが空っぽになる
```

## 2. ペルソナ

### 2.1 Phase 1: アカウント担当者（営業マネージャー）

- 担当顧客の業界動向を調査し、アカウントプランを策定する人
- 「この顧客に対して当社は何を提案すべきか」を考える立場
- 現実には調査に時間がかかり、プランの質にバラつきが出る

### 2.2 Phase 2: COO / 事業部長

- Salesforce提案時の最終提案先（CEOは投資の承認者）
- **軸足は収益寄り**。トップラインを上げることが至上命題
- 全ての情報を最終的に **「¥」に換算** して考える

| 役職 | 今のデモ環境との距離 | 理由 |
|---|---|---|
| CEO | 遠い | 抽象度が高すぎて直接的なデータ接点が少ない |
| CFO | 遠い | 原価・財務データはERPの領域 |
| **COO / 事業部長** | **近い** | サプライチェーン・品質・需要が全部関わる |

### 2.3 COOの日常的な問い

```
月曜朝の事業部会議の前に知りたいこと：
- 今月の需要計画に対してサプライチェーンにリスクはないか → 守り
- 品質問題で炎上しそうな案件はないか → 守り
- パイプラインの状況（来期以降の見通しは大丈夫か）→ 守り/攻め
- 市場機会はどこにあるか、自社はアラインできているか → 攻め
- 先週決定した対策の進捗は？ → フォロースルー
```

## 3. 外部データソース

### 3.1 設計方針

「業界コンテキスト」と「企業コンテキスト」を分離する。

```
業界コンテキスト（公的データから自動取得）
  ├ 市場規模・成長率
  ├ 政策・規制動向
  ├ 業界課題
  └ 競争環境
  → 実データをそのまま使える。誰に見せても問題なし。
  → Accountの業種に紐づく「マクロ環境」として全顧客に共通提供。

企業コンテキスト（手動 or AI支援で作成）
  ├ 顧客の経営戦略・中期計画
  ├ 投資方針
  ├ 固有の経営課題
  └ BPSとの取引履歴・関係性
  → デモでは手動作成。実環境ではSPEEDA等のAppExchange連携で自動化。
```

**デモでの語り方：**
> 公的データの自動取得で業界コンテキストのベースを作り、SPEEDAのようなAppExchangeパートナーでリッチ化する。どちらか一方ではなく、組み合わせがベスト。

### 3.2 日本の公的データソース

| ソース | 形式 | API | 更新頻度 | デモ適合度 | 用途 |
|---|---|---|---|---|---|
| **e-Stat（政府統計API）** | JSON/XML | REST API（無料登録） | 年次 | ★★★★★ | エネルギー統計の定量データ |
| **IRENA PxWeb** | CSV/JSON | PxWeb API（認証不要） | 年次 | ★★★★★ | 国別×技術別の再エネ容量・国際比較 |
| **FITポータル** | Excel | なし | 四半期 | ★★★★ | 都道府県別の再エネ導入量・買取金額 |
| **エネルギー白書HTML** | HTML | なし | 年次 | ★★★★ | 政策動向テキスト（章節ごとに個別HTML） |
| **GX政策文書（経産省）** | PDF/HTML | なし | 不定期 | ★★★ | GX投資150兆円ロードマップ等 |
| **環境省脱炭素ポータル** | PDF/HTML | なし | 不定期 | ★★ | 脱炭素先行地域リスト |

**最推奨: e-Stat API + IRENA PxWeb API**
- 両方ともREST APIでJSON取得可能
- e-Stat: 総合エネルギー統計（統計ID: 00551010）に1990-2024年の34年分
- IRENA: 200カ国以上の再エネ容量・発電量データ
- デモの「外部データ自動取得パイプライン」として最適

**デモで使える具体的数値：**
- 2030年再エネ電源構成比率目標: 36-38%
- 太陽光2030年目標: 14-16%（現状のほぼ倍増）
- 洋上風力2030年目標: 5.7GW
- 蓄電池2030年国内市場: 約2兆円規模（GX投資戦略）
- 脱炭素先行地域: 45都道府県133自治体が選定済

### 3.3 海外の公的データソース

| ソース | 対象地域 | API | 取得容易性 | 用途 |
|---|---|---|---|---|
| **World Bank Open Data** | 全世界 | REST v2（無料・キー不要） | ★★★★★ | 国別エネルギー指標。動作確認済み |
| **EIA（米国エネルギー情報局）** | 米国 | REST v2（無料登録） | ★★★★★ | 州別×エネルギー源別の発電データ |
| **Eurostat** | EU | REST（キー不要） | ★★★★★ | 国別再エネシェア |
| **NREL** | 米国 | REST（無料登録） | ★★★★★ | 座標ベースの太陽光発電ポテンシャル |
| **IEA** | 全世界 | 限定的 | ★★ | 一部無料データセットあり |

**最推奨: World Bank Open Data API**
- 完全無料、APIキー不要
- 全世界の国別エネルギー指標をJSON取得
- 実際にタイの再エネ電力比率を取得し動作確認済み
  - `api.worldbank.org/v2/country/THA/indicator/EG.ELC.RNEW.ZS?format=json`
  - → 2015年(13.14%) → 2021年(19.93%) の時系列データが正常返却

### 3.4 SPEEDA等の商用データとの位置付け

| ソース | SPEEDAが提供する情報 | 公的データで代替可能な範囲 |
|---|---|---|
| 業界概要 | ✅ 構造化済み | △ 白書テキストから抽出が必要 |
| 市場環境 | ✅ 定量データ豊富 | ✅ e-Stat/IRENA/World Bankで代替可能 |
| 競争環境 | ✅ 個社データ含む | △ 業界レベルは白書から可能、個社は限定的 |
| 業界課題 | ✅ 構造化済み | △ 政策文書からLLMで抽出可能 |

## 4. Phase 1: アカウントプラン策定支援

### 4.1 概要

Accountレコードページに配置するLWC。アカウント担当者がアカウントプランを策定する際に、外部データ＋SF内データを統合してAIが支援する。

### 4.2 機能

| 機能 | 内容 | AIの役割 |
|---|---|---|
| **業界コンテキスト表示** | 外部APIから取得した業界データをカード表示 | データ取得・要約 |
| **アジェンダ案の提示** | 業界コンテキスト + SF内データ（商談/ニーズ/品質）からアカウントプランのアジェンダ案を生成 | 情報統合→アジェンダ生成 |
| **戦略アライメント分析** | 顧客の戦略軸 × BPS製品/実績のマッチング | ギャップ分析 |
| **アカウントプランPPTレビュー** | PPTアップロード → Lambda + Claude Vision → テキスト抽出 → 内容のレビュー・改善提案 | PPT解析→レビュー生成 |
| **アクティビティレコメンデーション** | プランに基づく次のアクション提案（誰に会うべきか、何を提案すべきか） | 行動計画生成 |

### 4.3 アカウントプラン策定のフロー

```
① 業界コンテキストの自動提供
   外部API（e-Stat/IRENA/World Bank）→ Lambda定期取得 → Industry_Intelligence__c
   → Account.Industry と紐付けてLWCに表示

② 顧客固有コンテキストの作成支援
   AIがSF内データ（商談履歴/ニーズカード/品質/契約）を統合
   + 業界コンテキストを加味
   → 顧客戦略プロファイルのドラフトを生成
   → 担当者がレビュー・編集して確定 → Account_Strategy__c に保存

③ 戦略アライメント分析
   顧客戦略プロファイル × BPS製品ポートフォリオ × 商談実績
   → 「どこに攻めるべきか」「どこがギャップか」を可視化

④ アジェンダ案の提示
   ①②③の情報を統合 → アカウントプランのアジェンダ案をAI生成
   例: 「関東広域エネルギー公社 FY2027 アカウントプラン」
     1. GX2030への対応（分散型電源→既存SA拡大提案）
     2. AI需要予測の新規提案（SW-100拡張、¥1.2億機会）
     3. レジリエンス強化（蓄電システムのクロスセル）
     4. 設備IoT監視の導入加速（進行中DW商談のフォロー）

⑤ アカウントプランPPTのレビュー
   担当者がPPTを作成→LWCにアップロード
   → 既存Lambda（bps-demo-pptx-extractor）でテキスト抽出
   → AIがレビュー（業界コンテキストとの整合性、抜け漏れ、提案の強化ポイント）

⑥ アクティビティレコメンデーション
   プラン確定後 → 具体的なアクションリスト生成
   → Task/Event として起票可能
```

### 4.4 データモデル

```
Industry_Intelligence__c（新規 — 業界単位）
  ├ Industry__c: Text — 業界名（再生可能エネルギー、製造業等）
  ├ Category__c: Picklist — 政策動向 / 市場環境 / 競争環境 / 業界課題
  ├ Region__c: Picklist — 日本 / ASEAN / EU / 北米 / グローバル
  ├ Summary__c: LongTextArea — LLMが要約したテキスト
  ├ Key_Metrics__c: LongTextArea — 主要定量データ（JSON形式）
  ├ Source_URL__c: URL — 原文へのリンク
  ├ Source_Name__c: Text — 情報源名（e-Stat / IRENA / World Bank等）
  ├ Published_Date__c: Date — 原文の公開日
  └ Retrieved_Date__c: Date — 取得日

Account_Strategy__c（新規 — 企業単位）MD→Account
  ├ Market_Context__c: LongTextArea — 顧客の市場環境
  ├ Business_Strategy__c: LongTextArea — 経営戦略・中期計画
  ├ Investment_Focus__c: LongTextArea — 投資の方向性
  ├ Key_Challenges__c: LongTextArea — 経営課題
  ├ Regulatory_Drivers__c: LongTextArea — 規制・政策の影響
  ├ Strategic_Alignment__c: LongTextArea — BPSとの戦略アライメント分析結果
  ├ Action_Plan__c: LongTextArea — アクションプラン
  ├ Source__c: Text — 情報源（中計、IR、ニュース等）
  ├ Fiscal_Year__c: Text — 対象年度
  ├ Status__c: Picklist — ドラフト / レビュー中 / 確定
  └ Updated_Date__c: Date — 最終更新日
```

## 5. Phase 2: 経営ブリーフィング

### 5.1 概要

ホームページに配置するLWC。COO/事業部長が「守り（リスク）」と「攻め（市場機会）」を俯瞰し、意思決定→オペレーション起票→フォロースルーまでを一気通貫で行う。

**Phase 1のアカウントプランで蓄積されたデータが「攻め」セクションの情報源となる。**

### 5.2 基本フロー

```
俯瞰（守り: 供給/品質/需要 + 攻め: 市場機会）
  → 深掘り（AI分析 + 外部コンテキスト + 自社の強み）
    → 意思決定（選択肢 + トレードオフ + 収益インパクト）
      → オペレーション起票（RFQ/監査/プロジェクト/タスク等）
        → フォロースルー（次回ブリーフィングで進捗表示）
```

### 5.3 従来のダッシュボードとの決定的な違い

| 従来のダッシュボード | 経営ブリーフィング |
|---|---|
| 数字とグラフを**並べる** | 異常と示唆を**語る** |
| 見る人が自分で解釈する | AIが領域横断で解釈済み |
| 全指標を網羅的に表示 | **異常があるものだけ**表示 |
| 「で、何をすべきか」不明 | 対策の選択肢＋トレードオフ付き |
| データソースが独立 | 供給リスク×品質×需要が**連動して語られる** |
| 見て終わり | 意思決定→起票→進捗追跡まで一気通貫 |

### 5.4 UIの方向性

demandImpactAnalysisのWhat-If UIが方向性として近い。チャットではなく、構造化されたカードUI。

- 日本の経営者はチャットUIを使わない → Agentforceチャット直接は不採用
- What-If的な「前提設定→分析実行→結果表示→アクション」の構造が受けがいい

```
│  │   ⚠ 2件  │ │   🔴 1件  │ │   ● 順調  │            │
│  └──────────┘ └──────────┘ └──────────┘            │
│                                                     │
│  ┌──────────┐                                       │
│  │ 市場機会  │                              ← 攻め   │
│  │   🟢 3件  │  ← Phase 1のアカウントプランから集約    │

│                                                     │
│  [前回の決定事項: 2件進行中]              ← フォロースルー│
└─────────────────────────────────────────────────────┘
```

**各セクションのデータソース：**

| セクション | 表示内容 | データソース |
|---|---|---|
| 供給リスク | キャパシティ逼迫部品（影響金額付き） | Supplier_Capacity__c + Sales_Agreement + BOM展開 |
| 品質状況 | 重大CAオープン件数、増加傾向サプライヤー | Corrective_Action__c + Supplier_Investigation__c |
| 需要見通し | 確定SA + DWパイプライン（加重）、前月比 | Sales_Agreement_Schedule__c + Revenue_Forecast__c |
| 市場機会 | 戦略アライメントギャップ上位、未対応ニーズ | **Account_Strategy__c** + Needs_Card__c + Opportunity_Summary__c |
| 決定事項 | 前回起票したレコードのステータス | ソースタグ付きのRFQ/Audit/DP/Task等 |

全てのアラート・数値に **影響金額（¥）** を併記する。

### 5.6 深掘りシナリオ

#### 供給リスクの深掘り

AIが以下を統合して選択肢を生成：
- DemandImpactController: BOM展開→部品レベル需要→キャパ突合
- Revenue_Forecast__c: DWパイプラインの加味
- Supplier_Capacity__c: 代替拠点のキャパ余力

選択肢の例：

| 選択肢 | 効果 | リスク | 収益インパクト |
|---|---|---|---|
| A. 九州工場オフロード（短期） | 稼働率72%に低下 | 品質認定未完了、2ヶ月要 | 認定完了まで¥1,100万/月リスク残存 |
| B. 代替サプライヤー開拓（中期） | 依存度50%以下に | 新規認定6ヶ月、単価+8-12% | Q4以降の安定供給確保 |
| C. 需要側調整（交渉） | キャパ逼迫ピーク回避 | 顧客関係リスク | ¥3,200万の売上計上が1M遅延 |

→ [サプライヤー監査を起票] / [RFQを起票] / [担当者にタスク作成]

#### 品質アラートの深掘り

### 5.8 フォロースルー（決定事項の進捗追跡）

起票時にレコードにソースタグを付与：

```
起票されるレコード（RFQ__c / Supplier_Audit__c / Design_Project__c / Task 等）
  ├ Briefing_Source__c: Boolean = true
  ├ Briefing_Date__c: Date
  └ Briefing_Context__c: Text
```

次回ブリーフィング時に `Briefing_Source__c = true` でフィルタし、ステータスを表示。決定事項が完了するまで表示し続ける。

## 8. 未整理の論点

| # | 論点 | 備考 |
|---|---|---|
| 1 | 外部API取得の頻度・タイミング | Lambda定期バッチ（週次? 月次?）。デモではオンデマンドボタンでも可 |
| 2 | Industry_Intelligence__cの業種分類体系 | BPSの顧客は多業種。どの粒度で業界を定義するか |
| 3 | 戦略アライメントの「対応状況」判定ロジック | SA/DW/ニーズカードの有無で機械的に判定可能か、AI判定が必要か |
| 4 | 選択肢生成のプロンプト品質 | トレードオフが的外れだとオチが台無し。プロンプト設計が肝 |
| 5 | Phase 1→Phase 2のデータ連携 | Account_Strategy__cの集約クエリ設計 |
| 6 | 商談サマリカードの前提 | OpportunitySummaryBatchが安定稼働していると「自社の強み分析」の質が上がる |
| 7 | Siam Green Powerのデモデータ作成 | Account + Contact + 商談 + DW + SA等の一式。通貨はTHB |
| 8 | PPTレビューの差分 | 既存Lambdaはテキスト抽出のみ。レビュー生成はPrompt Template追加が必要 |
| 9 | Slack連携 | 「この所見をSlackに共有」は将来的に欲しいが初期スコープ外 |
| 10 | Agentforceとの役割分担 | アカウントプランのアジェンダ生成はAgentforce Topicにもなり得る |
