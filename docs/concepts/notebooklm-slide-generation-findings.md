## 1. 背景

OODAマネジメント関連のNotebookLM（30ソース）と、BPSデモ環境（ニーズカード・施策落とし込み機能）を組み合わせたCIO向けプレゼン資料を、NotebookLMのスライド生成機能で作成する試行を実施。

### 使用ツール

- **notebooklm-mcp-cli** (jacob-bd/notebooklm-mcp-cli v0.5.15): pip/pipxでインストール。Claude CodeからMCP経由でNotebookLMを操作
- **主要ツール**: `studio_create`（生成）、`studio_status`（進捗確認）、`studio_revise`（個別修正）、`notebook_query`（ソースへの質問）
- **デザインテンプレート参照**: bwPhD/awesome-notebookLM（NotebookLMスライド用プロンプト集）

## 2. 実験結果（3バージョン比較）

| ver | プロンプト特徴 | 結果 |
|---|---|---|
| v1 | メッセージ中心、情報粗い | グラフィカルすぎる。見出しサイズや構造がチグハグ |
| v2 | awesome-notebookLMの「Modern Newspaper」テンプレート適用 | デザインの一貫性は大幅改善。ただしインパクト重視でCIO向けには不適 |
| v3 | 原稿レベルで全テキストを書き下し＋デザイン指示 | 情報の粒度が保たれた。デザインはNotebookLM固有の問題が残る |

## 3. 効くこと・効かないこと

### 効くこと

| 要素 | 詳細 |
|---|---|
| テキストの忠実再現 | 原稿レベルで書き下せば、NotebookLMはテキストをほぼそのまま使う |
| デザインテンプレートの反映 | カラーパレット、タイポグラフィ比率、レイアウト方針は反映される |
| ソースの知見活用 | `notebook_query` で事前にソースから具体数字・事例を引き出し、プロンプトに埋め込むのが有効 |
| スライド種別の指定 | 「説明スライド」「デモ導入スライド」「メッセージスライド」と種別を明記すると構造が安定する |

### 効かないこと

| 要素 | 詳細 |
|---|---|
| スライド枚数の厳守 | 「8枚厳守」と指示しても守らないことがある |
| デザインの格調 | NotebookLM固有のグラフィック要素（アイコン、装飾）が入り、CxO向けの品質には届かない |
| デザインの一貫性（テンプレなし） | テンプレートなしだとスライド間でスタイルがバラつく |

## 4. NotebookLMへのプロンプト設計原則

### 4.1 NotebookLMの傾向

- メッセージ性に寄せたスライドを生成する傾向が強い
- 情報量が少ないプロンプトを渡すと、ソースから勝手に情報を追加・改変する
- グラフィカルな装飾を入れたがる

### 4.2 対策として有効だったプロンプト構造

```
冒頭: 「テキスト内容の改変・省略・追加は禁止。レイアウトとデザインのみ担当」と明記
全体ルール: 枚数、言語、カラー、フォント比率を指定
各スライド:
  - 種別（説明/デモ導入/メッセージ）を明示
  - 見出し → 本文（ブロック分割）→ 表 → 補足 の順序で構成を指定
  - CxOに不要な技術用語は事前に排除
```

### 4.3 情報量の目安

- メッセージだけ渡す（v1方式）→ NotebookLMが勝手に解釈・装飾する
- 原稿レベルで渡す（v3方式）→ テキストが忠実に反映される
- **原稿レベルが推奨**。NotebookLMに任せるのはレイアウトとデザインのみにすべき

### 4.4 ソースからの情報引き出し

`notebook_query` でソースに質問し、具体数字・事例を取得してからプロンプトに埋め込む手順が有効。
例: キーエンスのニーズカード制度（月2,000件、粗利80%条件、30社検証）やDeereのデータ事業（農薬90%削減等）の具体数字はすべてこの方法で取得した。

## 5. CxO向け資料への現実的なアプローチ

NotebookLMのスライド生成は「ラフの叩き台」としては有用だが、CxO向け品質にはデザイン面で届かない。

### 推奨ワークフロー

1. **NotebookLMのソースから知見を引き出す**（`notebook_query`）
2. **Claude Codeでスライド原稿を作成**（構成・テキスト・数字を詰める）
3. **NotebookLMでスライド生成**（叩き台として）
4. **PPTテンプレートに原稿を流し込んで仕上げる**（デザインは人間が担当）

### 将来的な改善可能性

- awesome-notebookLMのテンプレートをCxO向けにカスタマイズすれば、NotebookLMだけで完結できる可能性がある
- `studio_revise` による個別スライド修正は未検証（今回はスライド全体の再生成で対応）

## 6. notebooklm-mcp-cli セットアップメモ

```bash
# インストール
pipx install notebooklm-mcp-cli

# Claude Codeへの登録
nlm setup add claude-code

# 認証（ブラウザが開いてGoogleログイン → Cookie自動取得）
nlm login

# Claude Code再起動後に利用可能
```

- 認証はCookieベース。有効期限は約2〜4週間
- 35ツールがコンテキストを消費するため、使わないときは無効化推奨
- 非公式APIのためGoogleの仕様変更で動作しなくなるリスクあり
