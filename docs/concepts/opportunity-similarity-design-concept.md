# 商談類似度分析 × LLM活用 設計コンセプト

**目的**：営業担当者が商談を進める中で、組織内の類似商談・関連提案書から示唆を自動的に得られる仕組みを構築する
**作成日**：2026-03-24
**ステータス**：PoC動作確認完了（2026-03-24） — Agentforce Topic + 1アクション統合パターンで類似商談分析が動作

---

## 1. 背景と課題

### 1.1 解決したい問題

営業担当者が商談を進める際、組織内に蓄積された以下の知見を活用できていない：

- 過去・進行中の**類似商談**の進め方・成功/失敗パターン
- 類似商談に紐づく**提案書**の内容・アプローチ

### 1.2 なぜ難しいか — 商談データの本質的なブレ

商談データは入力者・商材によって大きくブレるため、生データ同士の直接比較では類似度を正しく測定できない。

| ブレの種類 | 具体例 |
|------------|--------|
| 活動登録の粒度 | 毎回詳細に記録する人 vs 要所だけの人 |
| 項目値の判断基準 | フェーズの進め方、確度の付け方が個人でバラバラ |
| 提案書の有無 | 添付する人・しない人、格納場所もバラバラ |
| 商材による違い | 単品売り vs ソリューション提案で商談の構造自体が異なる |

### 1.3 核心的な課題

**商談の「意味」を組織内で共通言語化する中間表現が必要**

商談を構成する情報（商品・活動・項目値・アカウント背景・提案書）を集約・抽象化し、比較可能な共通フォーマットに変換する仕組みがなければ、類似度測定も示唆抽出も機能しない。

## 2. アプローチ：中間表現の自動生成

### 2.1 ニーズカードとの対比

| | ニーズカード | 今回の中間表現 |
|---|---|---|
| 方向 | **分解・抽出**（商談→面談→ニーズカード） | **集約・統合**（商談＋活動＋商品＋アカウント→中間表現） |
| 粒度 | 個々の顧客の声を拾う | バラバラな情報を一つの意味ある表現にまとめる |
| 用途 | 製品側へのフィードバック | 営業側への類似商談・示唆の提供 |

### 2.2 中間表現の構造（検証結果に基づく暫定版）

Step 2の検証（4商談でLLM生成を実施）により、中間表現「商談サマリカード」の構造を帰納的に導出した。**構造化フィールド**（比較・フィルタリング用）と**自由記述フィールド**（意味的類似度用）の2層構造。

#### 構造化フィールド（列挙型 — ファセット検索・フィルタリング用）

| フィールド | 選択肢 | 用途 |
|-----------|--------|------|
| `deal_type` | 設備導入 / ソフトウェア・クラウド / Design Win・OEM / 保守・サービス / コンサルティング / その他 | クラスタの大枠分離 |
| `customer_segment` | 製造業 / エネルギー・ユーティリティ / 金融 / 公共 / 商社 / IT・テクノロジー / その他 | 業種軸でのフィルタ |
| `deal_scale` | 小規模(〜500万) / 中規模(500万〜3000万) / 大規模(3000万〜1億) / 超大規模(1億〜) | 規模感での絞り込み |
| `product_categories` | 配列（設備・プラント / モジュール・部品 / 機器・装置 / ソフトウェア / コンサルティング / 保守・サポート） | 商品構成の比較 |
| `sales_motion` | 新規開拓 / 既存深耕 / 競合リプレース / 紹介・パートナー経由 / 入札・RFP | 営業アプローチの分類 |
| `decision_complexity` | 単独決裁 / 委員会・合議 / 入札 / 不明 | 意思決定構造 |
| `competitive_situation` | 競合なし / 競合あり(優位) / 競合あり(拮抗) / 競合あり(劣位) / 不明 | 競合状態 |
| `urgency_driver` | 規制・法令 / コスト削減 / 設備更新・老朽化 / 事業拡大 / 事故・障害対応 / 不明 | 案件の動機 |

#### 活動分析フィールド（構造化 — 営業プロセスの比較用）

| フィールド | 型 | 用途 |
|-----------|---|------|
| `sales_process_pattern` | 列挙（技術主導型 / 関係構築型 / PoC主導型 / 入札対応型 / ...） | 営業プロセスの類型 |
| `key_stakeholders` | 配列 `[{role, title, name}]` | 関与者の構造（誰と会っているか） |
| `activity_summary` | テキスト | 活動パターンの要約（ノイズ除去済み） |
| `activity_milestones` | 配列（時系列） | 商談進行の主要マイルストーン |
| `engagement_depth` | 列挙（深い / 中程度 / 浅い） | 顧客との接点の厚み |

#### 自由記述フィールド（テキスト — ベクトル類似度用）

| フィールド | 内容 |
|-----------|------|
| `customer_challenge` | 顧客が解決したい課題の要約（2-3文） |
| `our_value_proposition` | 当社の提案価値・差別化ポイント（2-3文） |
| `deal_narrative` | 商談のストーリー・流れの要約（3-5文） |
| `key_risks` | 主要リスク・懸念事項（箇条書き） |
| `lessons_or_insights` | 他の営業が参考にできる示唆 |

### 2.3 設計上の残論点

- 何を拾い何を捨てるかの判断基準が商材・業種・フェーズで異なる → **検証でLLMの文脈判断が有効と確認**
- 入力されていない情報からの推論・補完をどこまで行うか → **Sparseデータではnullが正解。無理な推論はしない**
- 集約結果が「比較可能」であるための共通軸の定義 → **構造化フィールドの列挙値で担保**

## 3. 提案書の取り込み

### 3.1 現実の格納場所

提案書は一般的にSalesforce Filesではなく、SharePoint / OneDrive 等の外部ストレージに格納される。中間表現に統合するには外部ストレージからの取得経路が必要。

### 3.2 技術的制約（現時点で棚上げ）

以下の制約は存在を認識しつつ、初期検証では対象外とする：

| 制約 | 内容 |
|------|------|
| PPTの直接読み取り | Salesforce / Data CloudではPowerPointを直接パースできない |
| 図表の意味解釈 | スライド内のチャート・図の意味をLLMで解釈するのは現時点で困難 |
| 外部ストレージ連携 | SharePoint → Salesforce/Data Cloud への接続経路の構築 |

### 3.3 段階的アプローチ（案）

1. **Phase A**：提案書なしで中間表現の検証（商談データのみ）
2. **Phase B**：Salesforce Filesに格納されたテキスト抽出可能なファイル（PDF等）を対象に追加
3. **Phase C**：外部ストレージ連携 + PPT対応

### 2.5 中間表現の入力ソース（活動のノイズ処理含む）

```
Opportunity（商談）
  ├─ 標準項目（フェーズ、金額、確度、クローズ日等）
  ├─ カスタム項目
  ├─ OpportunityLineItem（商品）
  ├─ Task / Event（活動）← ノイズ含む（下記参照）
  ├─ Account（顧客背景：業種、規模、所在地等）
  ├─ Contact / OpportunityContactRole（関係者）
  └─ 提案書（※Phase B以降）
        ↓ LLMで集約・構造化（ノイズ耐性あり）
  商談サマリカード（共通フォーマット）
```

#### 活動データのノイズ問題と対処

実環境の活動データには以下のようなノイズが混入する：

| ノイズの種類 | 例 |
|---|---|
| 接待・社交 | ゴルフコンペ、会食、懇親会 |
| 季節行事 | お中元・お歳暮手配、年賀状、年度末挨拶 |
| 低情報量の記録 | 電話（不在）、カタログ送付 |
| 別案件との混在 | 定例訪問で複数案件を同時に扱う |
| 社内タスクの紐付け | 社内戦略会議、見積システム入力 |

**検証結果：LLMはこれらのノイズを適切に処理できた。**

重要な発見として、**ノイズの一部は「関係構築スタイル」の指標として活用できる**：
- ゴルフ・会食が多い → `engagement_depth: 深い`、`sales_process_pattern: 関係構築型` に反映
- 電話不在・カタログ送付が初期に集中 → 「初期段階での接点構築に苦労した形跡」として `activity_summary` に反映
- 社内会議・見積入力 → 適切にスキップ

つまり、活動のノイズを事前にフィルタリングするのではなく、**LLMに全データを渡して文脈の中で取捨選択させる**のが正解。

## 4. 類似度の測定手法

検証の結果、**ハイブリッド方式が有効**であることが示唆された。

| アプローチ | 概要 | 検証での評価 |
|------------|------|-------------|
| RAG（ベクトル検索） | 中間表現をベクトル化しセマンティック検索 | 自由記述の類似度測定には有効だが、構造的な差異（deal_typeの違い等）を見逃す |
| 構造比較 | 構造化フィールドでファセットマッチング | deal_typeだけで4クラスタが分離できた。大枠の絞り込みに有効 |
| **ハイブリッド（推奨）** | 構造化フィールドで大枠を絞り、自由記述のベクトルでランク付け | **検証結果が支持** — 下記参照 |

#### ハイブリッド方式の根拠（A-1 vs A-3の比較から）

同クラスタ（Cluster A: 再エネ設備導入）内のA-1とA-3は：
- **構造化フィールドでは高類似**（deal_type, customer_segment, deal_scale, product_categories がほぼ一致）
- **自由記述では明確な差異**（営業プロセス・競合状況・関係の深さが全く異なる）

| 比較軸 | A-1（アライドパワー） | A-3（中部電設） |
|---|---|---|
| `sales_motion` | 既存深耕 | 競合リプレース |
| `competitive_situation` | 競合なし | 競合あり(拮抗) |
| `sales_process_pattern` | 関係構築型 | 技術主導型 |
| `engagement_depth` | 深い | 中程度 |
| `customer_challenge` | 新拠点に設備が必要（シンプル） | 既存ベンダーからの乗り換え（複雑） |
| `our_value_proposition` | 実績＋信頼関係 | 保守TCOでの逆転 |
| `lessons_or_insights` | 既存実績の横展開 | 既存ベンダー不満客へのアプローチ |

つまり **「何を売るか」は同じでも「どう売るか」が全く異なる** — この差異は構造化フィールドの一部（sales_motion等）で大枠を捉え、自由記述のベクトル類似度で精密にランク付けする必要がある。

#### Sparseデータの限界（A-4の結果から）

商談名と金額のみのA-4（ノヴァテック 新R&Dセンター 電源設備）では：
- `deal_type: 設備導入` は商談名から推論できた
- しかし `customer_challenge`, `our_value_proposition`, `lessons_or_insights` はすべて **null**
- 類似度測定の対象にはなるが、**示唆を提供する側としては機能しない**

→ 中間表現の品質は入力データの品質に直結する。これは「入力を促す仕組み」の必要性を示唆する（フィードバックループへの接続点）。

## 5. フィードバック・効果測定

類似商談の示唆を提供して終わりではなく、その示唆が実際に役立ったか・行動につながったかを捕捉する仕組みが必要。

**検討すべき論点：**

- 示唆を受けた営業の行動変容をどう測定するか
- 示唆に対するフィードバック（役に立った / 的外れだった）の収集方法
- フィードバックを中間表現や類似度測定の精度改善に還元するサイクル

> この領域はまだ構想段階。中間表現と類似度測定の検証が進んだ後に具体化する。

### Step 2 実施結果（2026-03-24）

4商談（A-1, A-3, B-1, C-1）＋Sparse 1商談（A-4）で中間表現を生成。

#### 主要な発見

**1. 構造化フィールドだけでクラスタの大枠は分離できる**

`deal_type` の1フィールドで4クラスタがほぼ分離：
- Cluster A: 設備導入
- Cluster B: ソフトウェア/クラウド
- Cluster C: Design Win/OEM
- Cluster D: 保守/サービス

**2. 自由記述が「営業アプローチの類似度」の鍵**

同クラスタ内（A-1 vs A-3）でも `customer_challenge` や `deal_narrative` に明確な差異が出る。「何を売るか」が同じでも「どう売るか」の違いを自由記述が捉えており、これがベクトル検索の対象。

**3. 活動データのノイズはLLMが文脈で処理できる**

ゴルフ・会食等のノイズを除外するのではなく、「関係構築スタイル」の指標として中間表現に反映。電話不在の連続は「初期の接点構築の苦労」として解釈。**事前フィルタリングよりLLMに全データを渡して文脈判断させるのが有効**。

**4. 活動分析フィールドの追加が有効**

初回プロンプトでは活動情報が `deal_narrative` に溶け込むだけだったが、`sales_process_pattern`, `key_stakeholders`, `activity_milestones`, `engagement_depth` を明示的に追加したことで、営業プロセスの比較精度が大幅に向上。

**5. Sparseデータの限界は明確**

A-4（商談名と金額のみ）では `deal_type` の推論は可能だが、`customer_challenge` 等の核心フィールドはすべてnull。中間表現の品質は入力データの品質に直結する。

### Step 3-4 実施結果（2026-03-24）

全14商談でサマリカードを生成し、クラスタ間の類似度関係を検証。

#### 構造化フィールドによるクラスタ分離の評価

| deal_type | 対象商談 | 正しく分離できたか |
|---|---|---|
| 設備導入 | A-1, A-2, A-3, A-4, **C-2** | **C-2がCluster Aに混入** — メガソーラーは「設備導入」だが商社経由の大型調達 |
| SW/クラウド | B-1, B-2, B-3, B-4 | **完全一致** |
| DW/OEM | C-1, C-3 | C-2が欠落 |
| 保守/サービス | D-1, D-2, D-3 | **完全一致** |

→ deal_type単独では13/14（93%）の精度。C-2のような境界ケースは自由記述のベクトル類似度で補完する必要あり。

#### クラスタを横断する意外な類似パターン

**発見1: sales_motionの横断**

A-3（設備導入）とD-3（保守切替）はクラスタが異なるが、ともに「中部電設 × オメガからの競合リプレース」。アカウント戦略の文脈で関連付けるべき商談ペア。

**発見2: urgency_driverの横断**

B-3（金融の設備保全クラウド）とD-3（エネルギーの保守切替）は全く別クラスタだが、「事故・障害を契機とした緊急度の高い案件」という共通項がある。「障害対応を動機にした案件の進め方」という切り口で参考になる。

**発見3: 規制駆動のクラスタB内パターン**

B-1（補助金）, B-2（Scope 3）, B-4（GHG報告義務化）は全てurgency_driverが「規制/法令」。エネルギー管理PF案件の主要な動機が規制対応であることが構造的に確認できた。

#### Sparseデータの最低ライン

| データレベル | 例 | 生成可能なフィールド | 実用性 |
|---|---|---|---|
| Description + Products + Tasks | A-1, A-3, B-1, B-4 | 全18フィールド | **十分** |
| Description のみ | B-2, B-3 | 10/18フィールド | **最低限機能** |
| 商談名 + 金額のみ | A-4, D-2 | 5/18フィールド | **実用性低い** |

→ **Descriptionがあるかどうかが品質の分水嶺**。

#### 類似度関係の検証結果

```
✅ 期待通りの高類似度（同一クラスタ内）:
  A-1 ≈ A-2 ≈ A-3  → deal_type + product_categories 一致
  B-1 ≈ B-2 ≈ B-3 ≈ B-4 → deal_type 完全一致 + urgency_driver共通
  C-1 ≈ C-3 → deal_type + process_pattern 一致
  D-1 ≈ D-3 → deal_type 一致

✅ 想定外だが有用な中程度の類似（クラスタ横断）:
  A-3 〜 D-3 → 同一顧客 × 競合リプレース（アカウント戦略の文脈）
  B-3 〜 D-3 → urgency_driver（事故/障害対応）が共通
  B-1 〜 B-2 〜 B-4 → urgency_driver（規制/法令）のサブクラスタ

✅ 期待通りの低類似度（クラスタ間で異なる）:
  A-1 ≠ B-1, A-1 ≠ C-1, D-1 ≠ C-1 → 全て明確に分離
```

## 7. スケーラビリティの課題と次の方向性（2026-03-24 検討）

### 7.1 14件だから上手くいっている問題

Step 4の検証結果は14件のテストデータに対しては良好だったが、実環境（商談1,000件〜）ではいくつかの構造的な課題が予見される。

| 課題 | 14件では | 1,000件では |
|---|---|---|
| deal_typeでの絞り込み | 各グループ3-4件 | 各グループ200-300件。絞り込みとして不十分 |
| 類似度ランキング | TOP3が明確 | 類似度0.85と0.82の差に意味があるか不明 |
| 候補の数 | 2-3件で管理しやすい | 「類似商談30件」は見る気が起きない |
| 真に参考になる商談 | ほぼ全件が参考になる | 類似度が高いほど「既に知っている情報」と被りやすい |

### 7.2 「類似商談を探す」から「文脈に応じた検索」へ

固定的な類似度ランキングではなく、**営業が自然言語で「今知りたいこと」を表現し、それに応じた検索を行う**方が実用的。

```
例1: 「グリーンテックジャパンが競合にいる商談で、勝った事例は？」
  → competitive_situation + deal_narrative を検索

例2: 「設備選定委員会で決まる案件で、うまくいった事例は？」
  → decision_complexity = 委員会/合議 でフィルタ + 受注済み

例3: 「今の商談と似た状況だけど、違うアプローチで進めた商談は？」
  → customer_challenge が類似 + sales_process_pattern が異なる
```

これは**Agentforce**の領域。中間表現を検索基盤として、Agentforceが自然言語→構造化クエリ＋ベクトル検索を組み合わせて実行する構成。

### 7.3 Agentforceでの実現に必要な検証

| やりたいこと | Agentforceで必要な機能 | 見通し |
|---|---|---|
| 自然言語→構造化フィールドでフィルタ | Topic + Action でSOQL実行 | おそらく可能 |
| 自然言語→自由記述のベクトル検索 | Data Cloud Search Index or 外部ベクトルDB | 要検証 |
| 検索結果から示唆を生成 | Prompt Template で結果を要約 | おそらく可能 |
| 複数Actionの組み合わせ（フィルタ→検索→要約） | Agent のオーケストレーション | **要検証** |

→ **次のステップはAgentforceの技術検証に移行**。中間表現の設計は固まったため、「それをAgentforceからどう使えるか」のPoCに進む。
