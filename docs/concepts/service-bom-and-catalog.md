# サービスBOM (sBOM) × サービスカタログ設計 構想

**目的**：本番のIoT異常検知 → 商談機会推定パイプラインで、Gemini に「根拠のある明細見積」を出させるための上流マスタ設計
**作成日**：2026-04-23
**ステータス**：未着手（デモ版はプロンプトのガイドラインで金額範囲を縛る方式で対応中）

---

## 1. なぜ必要か

現在の Product Engineering Agent は、アラート生成時に「納入価格 × 0.6」のような粗い比率で商談機会金額を推定している。1件のIoTアラートにつき ¥108M（納入¥1.8億に対して60%）が計上され、業務的にはほぼ製品リプレース級の金額になってしまう。

本番運用でLTV予測や経営KPIに使うなら、以下が必要：

- **根拠のある明細見積**（部品費 + 工賃 + 保守契約 + アップグレード の分解）
- **整備されたマスタからの参照**（LLMの推測に依存しない）
- **補正可能な学習ループ**（実績との乖離で精度向上）

これには現在の **製造視点の BOM** だけでは不足で、**サービス視点のBOM（sBOM）+ サービスカタログ** が別途必要になる。

## 2. BOM の種類と現状整理

| 種類 | 視点 | 構成単位 | 現プロジェクトでの実装 |
|------|------|---------|--------------------|
| eBOM (Engineering BOM) | 設計 | 図面・仕様 | `BOM_Header__c` 以下で一部カバー |
| mBOM (Manufacturing BOM) | 製造 | 組立工程・調達部品 | `BOM_Part__c.Unit_Cost__c` 等で製造単価を保持 |
| **sBOM (Service BOM)** | **保守・サービス** | **現場交換単位・工賃・保守メニュー** | **無し（本構想で追加）** |

mBOM/eBOM は「この製品をどう作るか」の視点。sBOM は「この製品をどう維持するか」の視点で、**交換粒度・サイクル・工賃・アップセルオプション** を保持する別マスタになる。

## 3. sBOM に必要なデータモデル（案）

### 3.1 Service_Component__c（保守対象コンポーネント）

現場で交換する最小単位（製造部品とは粒度が違う）。

| フィールド | 型 | 用途 |
|----------|-----|------|
| Product2__c | Lookup | 対象製品 |
| Component_Name__c | Text | 冷却モジュール / 制御基板 / インバータ 等 |
| Replacement_Unit_Cost__c | Currency | 交換単価 |
| Labor_Hours__c | Number | 標準作業時間（時間） |
| Expected_Lifespan_Years__c | Number | 期待寿命 |
| MTBF_Hours__c | Number | 平均故障間隔 |
| Failure_Mode__c | Multi-picklist | 想定故障モード（LLMキーワード照合用） |
| Alert_Keywords__c | Text | アラート文言との照合用キーワード（「温度異常」「冷却」等） |

### 3.2 Service_Menu__c（サービスメニューカタログ）

| フィールド | 型 | 用途 |
|----------|-----|------|
| Menu_Name__c | Text | 定期点検プラン / リモート監視 / オンサイト訪問 |
| Product2__c | Lookup | 対象製品（製品別単価） |
| Unit_Price__c | Currency | 単価（¥/回 or ¥/年） |
| Billing_Cycle__c | Picklist | 回あたり / 年額 / 月額 |
| Included_Components__c | Multi-picklist | 対象コンポーネント |

### 3.3 Upgrade_Option__c（アップグレードカタログ）

| フィールド | 型 | 用途 |
|----------|-----|------|
| Option_Name__c | Text | FW v3.0 / 冷却システム液冷化 / 蓄電容量増強 |
| Product2__c | Lookup | 対象製品 |
| Unit_Price__c | Currency | 提案価格 |
| Target_Failure_Modes__c | Multi-picklist | どんな故障モードに対する根本対策か |

### 3.4 Labor_Rate__mdt（工賃レート）

カスタムメタデータ型で保持。サイト・技能レベル別の時間単価。

## 4. LLM への渡し方

アラート発生時、Vertex AI に以下をコンテキストとして渡す：

```
[Asset]
製品: EnerCharge Pro 蓄電システム (¥1.8億, 納入2024-08)
アラート: 冷却システム温度異常 (検知値 85℃ / 閾値 70℃)

[sBOM - 該当可能性コンポーネント]
- 冷却モジュール (交換単価 ¥3.5M, 作業5h, MTBF 30,000h)
- 温度センサー (交換単価 ¥0.2M, 作業1h)
- 制御基板 (交換単価 ¥1.2M, 作業3h)

[サービスメニュー]
- 年次定期点検 ¥0.8M/年
- リモート監視サービス ¥1.5M/年
- オンサイト緊急対応 ¥0.5M/回

[アップグレード]
- 液冷システム化 ¥12M (対象故障: 温度異常)
- FW v3.0 自動制御最適化 ¥3M

[工賃] 標準技能 ¥25,000/h
```

LLM は該当コンポーネントを選んで明細を組み立て、根拠のある見積（部品¥3.5M + 工賃¥0.125M + 液冷アップグレード¥12M + 5年監視契約¥7.5M = ¥23.1M）を生成する。

**効果**:
- 納入価格 × N% の粗い推定から、モジュール単位の明細見積へ
- Rationale が人間に理解可能・監査可能になる
- カタログ更新だけで単価・メニューが反映される（LLM再学習不要）

## 5. データ生成方針（現実の運用）

本番導入時、sBOM/サービスカタログの初期データは以下から組み上がる：

| ソース | 抽出対象 |
|--------|---------|
| 既存の保守契約書 | サービスメニュー単価、年次契約フォーマット |
| サービス部門の部品単価表 | Service_Component.Replacement_Unit_Cost |
| 過去修理オーダー | 実工数 → Labor_Hours のキャリブレーション |
| 設計部門のECM情報 | Upgrade_Option のカタログ化 |
| 故障解析レポート | Failure_Mode, Alert_Keywords の整備 |

## 6. 段階的ロードマップ

| Phase | 範囲 | 得られる価値 |
|-------|------|-------------|
| P0 (現状) | プロンプトで金額範囲を縛る | デモで違和感のない数字 |
| P1 | sBOM の Service_Component のみ追加 | 部品費ベースの根拠 |
| P2 | サービスメニュー + アップグレード追加 | 保守契約・アップセルの透明化 |
| P3 | 実績フィードバック（実際の商談化金額と推定の乖離分析） | LLM補正 / カタログ改訂 |
| P4 | Vertex AI LTV Projection（`asset-ltv-vertex-ai-projection.md`）に接続 | 継続率・追加商談確率も推論に組み込む |

## 7. 当面の回避策との接続

デモ版では **プロンプト内に業務相場のガイドライン**（納入価額の5-15%、内訳明記、IoT1件で製品リプレース級にならない）を埋め込む方式で対応する。このプロンプト改修は、本構想の P1 移行時には「sBOM参照ブロック」に置き換わる位置付け。
