# RAGにおけるドキュメントバージョン管理の課題

作成日: 2026-03-15
分類: エンタープライズアーキテクチャ論点整理

## 背景

Data Cloud RAGで外部ドキュメント（PDF、HTML等）をグラウンディングソースとして利用する場合、ドキュメントの更新・バージョン管理が大きな課題となる。Salesforce Knowledge記事のようにCRM内で完結するデータとは異なり、企業の実際のドキュメントは複数部門・複数システムで独立管理されており、全てをCRMに移行するのは非現実的。

## 現実のドキュメント構成

| ドキュメント種別 | 管理部門 | 管理基盤 | 更新頻度 |
|---|---|---|---|
| 社内FAQ | IT / 総務 | Knowledge or 社内ポータル | 随時 |
| 製品カタログ | マーケティング | CMS / DAM | 年1〜2回 |
| 技術仕様書 | R&D / 設計 | PLM / SharePoint | 製品リリース毎 |
| 価格表 | 営業企画 | Excel / SharePoint | 四半期 |
| 導入事例 | マーケティング | CMS / Web | 随時追加 |
| 規制・コンプライアンス文書 | 法務 | 文書管理システム | 法改正時 |
| 保守マニュアル | サービス | SharePoint / PDF | 製品更新毎 |

## 本質的な課題

### 1. ソースの多様性
- PDF、HTML、Excel、CRMデータが混在
- 管理基盤がSharePoint / S3 / Box / Google Drive / CMS等に分散
- コネクタの成熟度・対応状況がシステムごとに異なる

### 2. 更新タイミングの非同期性
- 部門ごとに独立して更新される
- Data Cloudの同期間隔（最短1時間）とのギャップ
- Search Indexの再ビルド時間（数分〜数十分）

### 3. バージョンの正確性（最重要）
- Data Cloudは追加は得意だが、旧版チャンクの選択削除APIが限定的
- 古い価格表や旧版仕様書でRAGが回答すると事故になる
- 新旧バージョンの矛盾する情報が同時にヒットするリスク

### 4. 責任分界
- ドキュメントを更新する部門 ≠ Data Cloudを管理するIT部門
- 「PDFを更新したらRAGにも反映される」という期待と現実のギャップ
- 誰がData Cloudへの取り込みとインデックス更新の正確性を保証するか

### 5. 正確性の監視・検証
- RAGが古い版を返さないことをどう継続的に検証するか
- 回答品質のモニタリング手法が未確立

## 現時点で考えられるアプローチ

### アプローチA: 全量洗い替え（シンプル）

```
[新版PDF格納] → 旧データストリーム削除 → 新規取り込み → Search Index再ビルド
```

- メリット: 確実にクリーン
- デメリット: ダウンタイムあり（再ビルド中は検索不可）
- 適用: 更新頻度が低い（四半期以下）ドキュメント

### アプローチB: Blue-Green切替（ダウンタイムゼロ）

```
[現行] Retriever_v1（有効）→ Prompt Templateが参照中
  ↓ 新版を並行構築
[新版] Retriever_v2（構築完了）
  ↓ Prompt Templateの参照先をv2に変更（CLIデプロイ）
[切替] v1を無効化・削除
```

- メリット: ダウンタイムゼロ
- デメリット: 一時的にリソース倍増、切替の自動化が必要
- 適用: 可用性要件が高い場合

### アプローチC: メタデータによるバージョン制御

```
[ドキュメントに version / effective_date メタデータを付与]
  ↓ Data CloudのDMOにバージョンフィールド
  ↓ Retrieverのフィルタで最新版のみ検索
```

- メリット: 履歴保持、柔軟
- デメリット: PDFにメタデータを付与する前処理が必要、フィルタの正確性に依存
- 適用: 構造化データ（CSV等）ベースの場合に有効。PDF単体では困難

### アプローチD: CRMデータへの移行（長期）

```
[PDF/HTML] → Salesforce Knowledge記事に移行
  ↓ CRM Connectorで差分同期（自動）
  ↓ 記事の公開/非公開でバージョン管理
```

- メリット: 最も安定、Salesforceネイティブ
- デメリット: 全ドキュメントの移行は非現実的（製品カタログ、技術仕様書等は無理）
- 適用: FAQ、社内ポリシー等のテキストベースドキュメント

## 理想的なアーキテクチャ（将来像）

```
[各部門のドキュメント管理基盤]
  SharePoint / S3 / Box / Google Drive / CMS / PLM
      ↓
[ドキュメント同期オーケストレーション層]  ← 現状ここが欠けている
  - 変更検知（Last Modified / Webhook / EventBridge）
  - バージョン判定（新版か差分か）
  - メタデータ付与（version, effective_date, doc_type, source_system）
  - 旧版の無効化・削除指示
  - 取り込みパイプラインの実行管理
      ↓
[Data Cloud]
  - ドキュメント種別ごとにデータストリームを分離
  - メタデータフィールドをDMOに保持
  - Search Index（種別ごと or Ensemble Retrieverで統合）
      ↓
[Ensemble Retriever]
  - 複数ソースを横断検索
  - フィルタでバージョンや有効期限を制御
  - 無関係な結果を自動除外（リランキング）
      ↓
[Prompt Template]
  - 回答にソースドキュメント名・更新日を必ず含める指示
  - 「この情報は〇〇（2026年3月版）に基づいています」
  - 矛盾する情報がある場合は最新のものを優先する指示
```

## MuleSoftの位置付け

「ドキュメント同期オーケストレーション層」はMuleSoftが埋める想定と思われるが、現状は：
- MuleSoft → Data Cloud のコネクタは存在する
- ただしドキュメントのバージョン管理・旧版削除の自動化まで含めたテンプレートは未確立
- 個別のインテグレーション開発が必要

## 提案・コンサル時の論点

1. **全ドキュメントをRAG対象にする必要があるか？** — スコープを絞ることで複雑さを大幅に減らせる
2. **更新頻度と正確性のトレードオフ** — 価格表（間違えると事故）vs 導入事例（多少古くても問題ない）でアプローチを分ける
3. **段階的導入** — まずKnowledge記事やFAQ（管理しやすい）でRAGの価値を実証 → 成功後にPDF/外部ドキュメントに拡張
4. **ガバナンス設計** — 誰が何のドキュメントの鮮度に責任を持つかを事前に定義
5. **回答の透明性** — RAGの回答には必ず「出典」と「更新日」を含める設計にし、利用者が鮮度を判断できるようにする
6. **監視** — 定期的にテストクエリを実行し、旧版情報が返らないことを検証する仕組み
