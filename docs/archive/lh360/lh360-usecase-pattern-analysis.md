# Local Headless 360 — ユースケースパターン分析

**作成日**: 2026-04-19
**位置付け**: Phase 3 β 作業。`docs/concepts/lh360-scope-reframing.md` §4 で方針提示した「ユースケースレベルの抽象化」の実施ログ。
**成果物**: ペルソナ × KPI × 行動指針 × タスク × A〜F パターンの 5 層対応表と、lh360 のカバレッジマップ。

---

## 方法論

top-down → bottom-up の往復で、「思いつきタスク列挙」を避け、業務構造から妥当性を持った棚卸しを行う。

1. **Step 1**: ペルソナ・ビジネスモデル宣言（スコープ固定）
2. **Step 2**: KPI ツリー展開（3 層、各層 3〜5 本）
3. **Step 2b**: 行動指針のグルーピング（P1〜Pn）
4. **Step 3a**: 各グループの**中粒度タスク**列挙（仕事の塊として認識される単位）
5. **Step 3b**: 中粒度タスクを **elementary task** に分解し A〜F パターン分類
6. **Step 4**: 行動指針グループとのカバレッジマップ作成

### 重要な前提（Q4 の洞察）

> パワーチャート作成のような一回限りの大タスクを分解すると、構成要素は「名刺・訪問記録の棚卸し」「顧客役員人事の Web リサーチ」など、**日常業務のスコープを広げたサブタスクの集合**になる。

これを受け、本分析では:

- **大タスクそのものは F 判定になりがち**（多段推論・全体構成・抽象的判断）
- しかし**elementary task は A〜E に収まる**傾向
- **同一の elementary task が複数の大タスクに再利用される**（→ 重複で収束する）
- **lh360 の使い所は elementary 単位**。大タスクは人間 or 上位 orchestrator が分解し、各 elementary は lh360 が担う

### elementary task の定義

- 1 つの意図 / 1 つの成果物に対応
- 複数の MCP tool を連鎖させる場合はある（= B / C パターン相当）
- ただし**人間の判断介在が挟まらない**（A〜E の完全自動可）
- 体感 30 秒〜5 分で完結する粒度

### 業界固有の設計原則（重要）

- **競合対策は「RFP で勝つ」のではなく「RFP 以前で排除する」のが基本**。このクラスの大型案件で公式 RFP 対決になった時点で、既に自社提案を顧客の要件書に埋め込めていない時点で劣勢。したがって**競合戦略は P3 提案策定ではなく P1-7 動向把握の中に組み込む**（先行ポジショニングが主戦場）
- **P3 に「社内調整」が吸収される**: 見積もり依頼、工程作成、工場リソース手配などの部門間調整は実質的に SAE の時間の大半を占める。これらは汎用「社内連絡」ではなく提案工程の一部として P3 に配置
- **P7 は案件外の横断コミュニケーションに純化**: SFA 操作・レポート作成・Legal/Finance との非案件系やり取りに限定。案件調整は P3/P4 側

## Step 3b: elementary task 分解 + A〜F 分類

### Step 3b-pre: A-F 判定表（分類ロジックの明文化）

elementary task を属人的に分類しないための判定基準。すべての elementary は以下の 6 属性で記述し、属性の組合せから A-F を機械的に判定する。

#### 判定に使う 6 属性

| 属性 | 値 | 説明 |
|---|---|---|
| **src** | `u` / `s` / `e` / `s+e` | 入力ソース: user発話 / SSoT / 外部 / 混合 |
| **op** | `fmt` / `corr` / `reason` / `classify` | 処理: 整形 / 相関 / 推論 / 分類判定 |
| **out** | `P` / `D` / `W-s` / `W-e` | 出力先: Present / Draft / SSoT書込 / 外部書込 |
| **hop** | 1 / 2 / 3 / 4+ | tool 呼び出し連鎖段数 |
| **trig** | `i` / `bg` | 起動: 対話的 / バックグラウンド |
| **absn** | `lo` / `mid` / `hi` | 抽象判断の要求度: 低 / 中 / 高 |

#### A-F の属性プロファイル

| Pat | 構造 | src | op | out | hop | trig | absn | Gemma 4 |
|---|---|---|---|---|---|---|---|---|
| **A** | SSoT 照会 + 整形 | u+s / s | fmt | P | 1 | i | lo | ◎ |
| **B** | SSoT + 外部相関 | s+e | corr | P | 2 | i | mid | ○ |
| **C** | SSoT 起点の外部書き込み | s / s+e | corr | D / W-e | 2-3 | i | mid | ○ |
| **D** | 外部 → SSoT 書き戻し | e | corr | W-s | 2 | i / bg | mid | ○ |
| **E** | バックグラウンド定期処理 | s (時間起点) | fmt / corr | P / D | 2-3 | **bg** | lo-mid | ◎ |
| **F** | 多段探索・深い推論 | s (FK 辿り多段) / s+e | reason | P | **4+** | i | **hi** | × |

#### 判定フロー（上から順に評価、最初にヒットしたものを採用）

```
1. trig=bg なら → E
2. hop≥4 または absn=hi なら → F
3. out=W-s（SSoT書込） なら → D
4. out=W-e（外部書込） または out=D（ドラフト） なら → C
5. hop≥2 かつ out=P なら → B
6. hop=1 かつ out=P なら → A
```

**修正履歴**: 初版では A の条件に `src=s` を含めていたが、外部のみ（src=e）の 1 hop fetch（例: 業界記事の取得・要約）が未分類になったため、`src` は判定から外し説明属性に留めた。A-F の構造は「hop × 出力タイプ」が本質で、src は手段の話。

#### 境界ケースと解決ルール

| ケース | 判定 | 根拠 |
|---|---|---|
| hop=2 で結果を Present のみ | **B** | 相関しているが書き込みなし |
| hop=2 で結果をメール下書きまで | **C** | Draft 出力がある時点で C |
| hop=3 以上でも FK 辿りが直線的（条件分岐なし） | **C or B** | hop だけで F 認定しない。「前段結果で次段の選択が変わる」かで判定 |
| hop=3 だが途中で**強い抽象判断**（例: 「重要度を評価」「最適な組合せを選ぶ」） | **F** | hop 数より判断の質で判定 |
| ユーザ発話起点だが実態は SSoT スキャン（例: "今日の未完 Task 見せて"） | **A** | hop=1 で Present のみ、trig=i |
| 外部の定期スキャン（例: 毎朝 Gmail の新規メール監視） | **D (bg)** または **E** | 結果が SSoT 書込なら D、通知/Present なら E |
| 提案書作成全体 | **F** | elementary ではなく集約タスク。elementary 分解した構成要素を A-E で拾う |
| 物理行動（実会議、訪問、電話） | `scope: out` | LLM で置換不可能。A-F 対象外 |

#### `scope: out` の扱い

- P7-10（経費精算）や物理行動は `scope: out` としつつ、**「もし lh360 で扱うなら」の仮分類は実施**（将来の拡張判断材料として）
- 仮分類欄には `(out) <Pat>` のように記録

#### lh360 の target 範囲

- **A / B / C / D / E** が lh360 の正規 target
- **F** は原則扱わない（orchestrator 層で分解するか、クラウド LLM へオフロード検討、scope-reframing §5 参照）
- F が出てきた elementary は「そのまま F」として残し、**F を構成する sub-elementary が A-E で書けるか**を Step 4 で評価する

### pilot で判明した判定ロジックの妥当性

- **判定フロー修正（`src=s` 制約削除）は正解**: P7 で `src=e` 起点の 1-hop タスク（領収書抽出、議事ドラフト等）が多く、src 制約を付けたままだと分類できない elementary が量産されていた
- **`absn=hi` を F 判定のトリガーにしたのは有効**: P1/P7 とも F 判定タスクは全て absn=hi。抽象判断が F パターンの真の識別子である裏付けが取れた
- **hop 数単体では F 判定しない**ルールも正解: P7 には hop=2-3 のタスクが多いが、多くは直線的連鎖（SFDC lookup → 書込等）で F にならない。条件分岐・判断介在が F の本質
