## 1. Gemma 4 の限界はどこから来ているのか

Phase 2 で観察された 4〜5 種類の失敗を、**コンテキスト量 / 推論能力 / 内在知識 / 訓練バイアス / 量子化ゆらぎ** で切り分けた。

### 観察された失敗と原因の対応表

| 観察された失敗 | 主因 | 補足要因 |
|---|---|---|
| 25 tools で tool_call フォーマット崩壊 | **① 出力フォーマット生成能力**（long-context 下での構造化出力生成） | コンテキスト総量 + 4bit量子化 |
| 多段 SOQL の連鎖推論ができない（「担当者誰？」） | **② 推論能力**（agentic depth / plan capability） | MoE active ~4B 相当 |
| `directory=.` のエラーループ（同じ invalid を 10 回リトライ） | **③ self-correction**（エラー文言→制約への推論反映） | tool schema 文字列の埋没 |
| 「システム上の制約により」と宣言して止まる・ハルシネーション | **④ 訓練バイアス**（RLHF の "謙虚に諦める" 癖） | ハルシネーション回避方向の over-training |
| 11 tools でも稀に崩壊 | **⑤ サンプリングゆらぎ** | 4bit 量子化で裾の確率が厚くなる |

### それぞれの本質

**① が最も重い**: 「context 容量（入れる側）」ではなく、「long-context 下で**正しい構造化出力を生成し続ける能力**」の限界。Gemma 4 の chat template では tool_call が特殊トークン列で表現されるが、入力が長くなるほど attention が散って、この特殊列の生成が不安定になる。MoE の active ~4B は、「長い context から的確な出力構造を生成し続ける」ほどのパラメータが裂けない。

**② は別物**: これは long-context とは独立で、tool 数を 11 に絞っても「担当者誰？」の多段 SOQL は組めない。Opportunity → AccountId → Account → Contacts という**外部キー辿り**の plan を、前段の結果を次段の引数に渡しながら展開する「agentic depth」は、active 4B では明確に足りない。

**③ は ② の親戚**: エラー文言 `Must be an absolute path` を読んで「自分の引数を絶対パスに変える」というのは、直前の observation から制約を抽出 → 自分の次の action に反映、という反射ループ。これも agentic 能力の一側面。

**④ は訓練由来**: Gemma 系は一般チャット向けに「無理な推論をせず協力を求める」方向に RLHF されている。これがエージェント用途では「詰まると止まる」癖として出る。

### 内在知識は問題の中心ではない

SOQL 構文・JSON・Salesforce の概念スキーマ等、**内在知識面は期待通り**。できないのは「知識の欠落」ではなく「手持ちの知識を連鎖させて行動する能力」。

### 実務的な含意

| 緩和策 | 効く領域 |
|---|---|
| MCP プロファイル分割（20 tools 未満） | ① |
| tool schema の簡素化・要約化 | ① |
| 自動リトライ + 温度再サンプル | ①⑤ |
| few-shot example を動的コンテキストに入れる | ②③ |
| 「諦めずに retry せよ」を system prompt に強調 | ④ |
| **MCP ラッパ層で引数を強制上書き / 既定値補完**（Phase 3 α(1) で採用） | ②③ を回避（根治せず "蓋") |
| **モデルを 70B 級 dense or reasoning系（o1-like）に上げる** | ②③④ を一気に改善 |

一番の根治はモデル格上げ。Gemma 4 26B A4B 4bit で agentic に多段推論を安定させるのは、アーキテクチャ的に無理筋の領域が多い。ローカル縛りを保つなら Qwen3 32B dense や Llama 4 の reasoning 派生の登場待ち、という状況。

### 補足: 「LLM に失敗を見せて自己修正させる」は Gemma 4 では期待できない（Phase 3 α(1) で確認）

Phase 3 α(1) の議論中に独立した観点として明確化:

- Phase 2 §8 TURN 3 の実測で、`directory` validation エラー（`Must be an absolute path` という**人間には自明なエラー文言**）を Gemma 4 に見せても **10 回同じ invalid 値でリトライし続け、最終的に諦めて止まった**
- これは単発の不運ではなく、③ self-correction（エラー文言 → 抽象制約抽出 → 次 action への反映）の能力不足と ④ RLHF「詰まったら諦める」バイアスの**合成結果**。active ~4B では構造的に回らない
- 一方、**正常系の tool chain**（Opportunity 検索 → カレンダー確認 → 候補提示の 2〜3 hop）は TURN 2 で 12 turns かけて完遂している。つまり「chain できる ≠ error から recover できる」が切り分けられた

**帰結**: Phase 3 α(1) で wrapper 層に引数ポリシー（defaults / overrides）を導入したのは、「LLM にエラーを見せて直してもらう」が実測で効かない以上、**"デプロイ時に分かっている知識を wrapper に押し込んで蓋をする" のが現実解**、という判断に基づく。この設計は:

- **モデル能力の改善にはならない**（LLM 側は自分の誤りに気付かない、次回も同じ誤りを出す）
- **デプロイ固有知識（固定 org、プロジェクトパス）にしか適用できない**（LLM に判断させる必要がある領域では使えない）
- **モデル格上げ時に剥がすべき候補**（Qwen3 32B dense 等なら self-correction が機能する可能性が高いので再評価する）

どんなエラー型なら「LLM に見せる」が機能しうるかの切り分け:

| エラーの性質 | 自己修正可能性 | 備考 |
|---|---|---|
| 引数名 typo / 不足フィールド | △〜○ | エラー文言がそのまま fix になる類型なら効く可能性 |
| 抽象的な形式制約（絶対パス、ISO 8601 等） | × | 実測 × |
| 値の意味的不整合（SOQL syntax 等） | △ | 単純な syntax なら直せるが複合条件で loop りやすい |
| ドメイン知識が必要な修正 | × | 連鎖推論が要求されるので詰まる |

## 2. ローカル LLM で汎用 agent は成立するか

### 認識した構造的制約

- **汎用（any-task / any-tool）かつフロンティア級の agentic behavior を Gemma 4 26B A4B 4bit で出すのは、現実的に無理**
- tool 数・多段推論・self-correction の 3 つが揃って求められる「Claude Code / Cursor 的な体験」は、active 4B 相当では構造的に出ない
- 消費者HW（M4 Max 128GB クラス）での最大は 70B dense 4bit 前後、これでもフロンティア（Claude Opus / GPT-5）との gap は明確

### ただし二項対立ではない

"フロンティア"と"ローカル"を 3 軸で考えると設計方針が変わる:

| 軸 | フロンティア向き | ローカル向き |
|---|---|---|
| **スコープ** | 汎用（any task） | ドメイン特化（sales / coding / RAG など） |
| **エージェント深さ** | 長い多段・自己修正 | 短い 2〜3 段 + 固定パイプライン |
| **tool 数** | 数百 OK | 10〜20 で厳選 |

つまり**「汎用 agent をローカルで」という目的設定自体が、HW・モデルの制約を無視した方向性だった**可能性がある。同じローカル LLM でも:

- 「Salesforce SSoT 操作 + カレンダー + メール」のように**業務スコープを絞る**
- multi-turn 探索ではなく、**"1 hop + 小さな自己修正"で完結するタスク**に限定
- 連鎖推論が必要な部分は**プロンプト側にチェックリスト・fewshot を仕込む**

という設計に寄せれば、Gemma 4 クラスでも実用ラインに乗る余地はある。Phase 2 の観察でも、TURN 2（訪問アポ提案）は 12 turns で完遂しており「2〜3 hop までは機能する」ことは実証できている。

### 結論

> **「ローカル LLM で汎用エージェント」は厳しい**
> **「ローカル LLM でスコープを絞ったエージェント」は現実的**

## 3. 一周してドメイン特化に戻る

### 皮肉な回帰

- Phase 1: Salesforce シナリオ特化の agent を作った（当時の名称 "Agentforce 2.0"）
- Phase 2: 「これ汎用基盤にできないか？」と一度剥がしにかかった → モデル能力の壁にぶつかり、スコープを戻すべきだと判明
- Phase 3 以降: 再び "Salesforce を中核としたドメイン特化 agent" に回帰する方が筋

### ただし一周は無駄ではない

1. **剥がしてみて初めて、何が domain-specific で何が general かが分離できた**。`environment.yaml` / `user_profile.yaml` で identity だけ注入し、base プロンプトは中立、という**設計パターンそのものは残す価値がある**。次に「Salesforce 以外の SSoT」を扱うときに、プロンプト書き直しではなく config 差し替えで済む土台になった。

2. **公式 MCP 優先・プロファイル切替・閾値の実測値**といった**運用知見は汎用で使える**。これは Phase 2 をやらなければ得られなかった。

3. **"Local Headless 360" という名前の再定義**: 単なる Phase 1 の続編ではなく、「ローカル LLM × 公式 MCP × Salesforce SSoT 特化 × 汎用化済みの基盤設計」という、Phase 1 より明確な意味を持った 2.0 になった。

つまり**名前が一周したのではなく、名前の中身が深まった**、というのが正しい整理。

## 4. 次にやるべき抽象化は "ユースケース" レベル

### 2 種類の抽象化の対比

Phase 2 の抽象化は**技術レイヤ**の話。いま議論すべきは**ユースケース構造**の話。性質が違う。

| | Phase 2 でやった抽象化 | これから議論すべき抽象化 |
|---|---|---|
| 対象 | SSoT / ドメイン語彙 / ツール接続 | タスクの**構造**・**形** |
| 目的 | SAP でも ServiceNow でも動くこと | 特定ユースケースから再利用可能パターンを抽出 |
| 単位 | 設定ファイル・プロンプト | タスクパターン（型） |

### ユースケースパターン分類案（A〜F）

| パターン | 構造 | hop 数 | Gemma 4 適性 |
|---|---|---|---|
| **A. SSoT 照会 + 整形** | Read→Present | 1 | ◎ |
| **B. SSoT + 外部相関** | Read×Read→Present | 2 | ○ |
| **C. SSoT 起点の外部書き込み** | Read→Correlate→Draft/Write | 2-3 | ○（Phase 2 で実証済） |
| **D. 外部 → SSoT 書き戻し** | External→Parse→Write | 2 | ○ |
| **E. バックグラウンド定期処理** | Trigger→Read→Draft | 2-3 | ◎（対話不要） |
| **F. 多段探索・深い推論** | Read→Read→Read→Reason | 4+ | ×（Gemma 4 では扱わない） |

### この抽象化の意義

ここまで抽象化して初めて、

- **ローカル LLM が得意なパターン（A〜E）を束ねた agent** = Local Headless 360 の現実的射程
- **F に相当するユースケース（「担当者誰？」のような多段探索）はそもそも取り扱わない or クラウド LLM に逃がす**
- **同じパターンなら新しいシナリオの追加コストが低い**（プロンプト+fewshot 差し替えで済む）

という設計判断ができるようになる。

### Phase 3 の方向性（暫定案）

1. **現実的ユースケースの棚卸し**: 営業・CS・マネジメント等、Salesforce ユーザが「ローカル LLM agent にやらせたい」ことを 10〜20 個列挙
2. **パターン分類**: 上記 A〜F に振り分け、F 相当は今回のスコープ外と明示
3. **パターン × MCP プロファイル の対応表**: 「B パターンは sales profile、D パターンは minimal + email、E パターンは cron + minimal」のように組合せを定める
4. **パターンごとの最小テンプレ作成**: system prompt fragment + fewshot + 推奨 MCP

こうすると、**「Local Headless 360 とは何か」が具体ユースケース集ではなく、"ローカル LLM で実用ラインに乗るタスクパターンの集合"** として定義できる。これこそが「名前の中身を深めた 2.0」の正体になる。

## 5. 補論: F パターンをクラウド LLM にオフロードする設計は成立するか

**問い**: ローカル LLM が自分で F を識別してクラウドに投げる設計は現実的か（今やる/やらないとは別として）。

**結論**: **成立する**。ただし「ローカルLLM 自身に自己判断させる」アプローチは不安定で、**オーケストレータ層の heuristics に寄せる**のが現実解。

### 成立する 3 つのアプローチ（難易度順）

#### 1. 事後検知型（最も確実）
Phase 2 の観察そのものがシグナルになる:

| 検知シグナル | 意味 |
|---|---|
| 同じ tool_call を N 回以上繰り返す | self-correction できていない |
| `finish_reason=tool_calls` で `tool_calls=None` | フォーマット崩壊 |
| turn 数が閾値（例: 6）を超えて stop せず | 多段で迷走 |
| assistant が「システム上の制約」等のハルシネーション定型句 | F パターンで詰まった兆候 |

→ オーケストレータが**失敗パターンを検知した時点で、同じ context をクラウド LLM に丸投げして続行**。これは「LLM に判断させる」のではなく「失敗の形を監視する」ので、モデル能力に依存せず動く。Phase 2 の実走ログはこの heuristics の設計素材としてそのまま使える。

#### 2. 事前ルーティング型（中堅）
**別の軽量モデル**（distilBERT クラス・ルールベースでも可）を入力段に置いてパターン分類:
- 「〜の担当者は？」「〜の理由は？」のような**関係照会**は F に寄りやすい
- 「〜の一覧」「〜を作って」のような**直接操作**は A-E に収まりやすい
- 「エンティティ数」「多段関係の示唆語（"そもそも"・"関連する"）」を特徴量に

メインの Gemma 4 に判定させると *the cat judges the dog* 問題（自分が扱えないものを認識する能力がそもそも不足）になるので、**別モデル**であることが重要。

#### 3. 明示的 escalation tool（補助）
`escalate_to_cloud_llm(reason, current_context)` というツールを LLM に渡す。Gemma 4 の「諦める」バイアスは**この設計では追い風**になる（詰まったら諦めてツール呼び出し、が自然に出る）。

**ただし難点**: Gemma 4 は本当にダメなときに"tool を呼ばずに stop"する癖があるので、これ単独では取りこぼす。→ 1 と組み合わせる前提。

### 設計上の難所

| 論点 | 課題 |
|---|---|
| **context 引き継ぎ** | 途中でクラウドに投げる時、system prompt / history / tool schemas / tool results をどう圧縮・転送するか（特に privacy 観点で「ローカル専用」だった情報の扱い） |
| **一貫性** | ローカル→クラウド→ローカルに戻すか、一度クラウドに渡したら戻さないか。途中でモデル切替すると style が変わる |
| **価値設計** | 「80% ローカル / 20% クラウドオーバーフロー」のような比率を崩さないと、そもそもローカル LLM にする意味が薄れる |
| **校正の難しさ** | ルーターがメイン LLM 自身より calibration が良くないといけない（ここが一番難しい） |

### 類似の先行事例

- **FrugalGPT**（研究）: cascade router で安いモデル→高いモデルに段階的にフォールバック
- **Ollama + Claude hybrid** 構成: コミュニティで実装例あり
- **MoE そのもの**: アーキテクチャ内部での routing（ただし能力の異種混合ではなく同等専門家の切替）
- **Claude Code 的な "subagent"**: 大元は強いモデル、subagent を弱いモデルに出すのは逆方向だが概念は近い

### 結論のまとめ

**成立する。しかも "pattern F 識別 → オフロード" は Phase 2 の観察と綺麗に結びつく設計**。ただし、

- **LLM の自己判断に頼らず、orchestrator 層の失敗パターン検知 + 軽量ルーター**の組合せが筋
- **明示的 escalation tool は補助として有用**（Gemma の諦め癖を設計に活かせる珍しいケース）
- 最大の設計コストは **context 引き継ぎ** と **privacy 境界**

Phase 3 でやるかどうかは別として、設計の選択肢としては確実に現実的なラインに入る。
