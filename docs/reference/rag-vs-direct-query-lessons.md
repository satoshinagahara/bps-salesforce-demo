# RAG vs 直接クエリの使い分け — イベントアンケートパイプラインでの学び

作成日: 2026-03-21

## 背景

イベントアンケート → Data Cloud → ニーズカード自動生成パイプラインの実装において、当初RAG（Retriever + Search Index）を全処理に適用したが、要件との適合性に課題があった。

## 学び: RAGは「検索」には強いが「集計・分析」には向かない

### RAGが適切なケース

- **大量データの中から関連情報を探す**（数千〜数万件のナレッジから「この質問に答えられる情報」を抽出）
- **対話型の質問応答**（「この顧客のアンケートで何と言っていた？」）
- **データの一部だけが必要**で、全件を見る必要がない場合

### RAGが不適切なケース（今回のケース）

- **全件を対象にした傾向分析**（「このイベントの全体的な反響は？」）
- **少数派の意見を確実に拾いたい**（ネガティブ意見はセマンティック検索で漏れやすい）
- **データ量が少ない**（30件程度ならLLMのコンテキストウィンドウに全件入る）
- **構造化データのフィルタリングが必要**（「この取引先の回答だけ」のような条件指定）

### 具体的に発生した問題

| 問題 | 原因 | 対処 |
|---|---|---|
| 取引先別の分析ができない | Retrieverはテキスト類似度で検索するため、company_nameでのフィルタリングができない | Data Cloud queryv2でメールアドレス/ドメイン指定で構造的に取得 |
| ネガティブ意見が拾えない | 「音声が途切れた」「UIがわかりにくい」等はアンケートのメインテーマと類似度が低く、チャンクとして返されにくい | 直接クエリで全件取得すれば漏れない |
| 複数取引先の意見が混在 | Retrieverはイベント全体のチャンクをまとめて返す | 取引先ごとにqueryv2で回答を分離してからLLMに渡す |

## 最終的な設計判断

| 処理 | 方式 | 理由 |
|---|---|---|
| **取引先別ニーズカード生成** | Data Cloud queryv2 + Prompt Template（Retriever不使用） | 取引先単位の正確なフィルタリングが必要 |
| **キャンペーン反響分析** | Retriever + Prompt Template → **要改善: queryv2に変更すべき** | 現状Retrieverだがネガティブ意見の取りこぼしが発生。全件取得が望ましい |
| **Agentforce対話型検索** | Retriever + Prompt Template（RAG本来の用途） | 「この顧客について教えて」のような自由な質問にはセマンティック検索が適切 |

## 一般化したガイドライン

```
「LLMに渡すデータをどう選ぶか？」

全件渡せるサイズ（〜数百件）？ → 直接クエリで全件取得 → LLMに渡す
  ↓ No
構造化条件でフィルタ可能？ → クエリでフィルタ → LLMに渡す
  ↓ No
自由テキストの意味検索が必要？ → RAG（Retriever）を使う
```

## Data Cloud連携の実装パターン比較

| パターン | 実装 | 特徴 |
|---|---|---|
| **queryv2直接** | Apex HTTP callout → `/ssot/queryv2` | 構造化フィルタ可能、全件取得、ステータスコード201に注意 |
| **Retriever（RAG）** | Search Index + Retriever + Prompt Template | セマンティック検索、チャンク単位、全件網羅性に欠ける |
| **Data Graph** | 複数DMO結合ビュー | リレーション結合、Retrieverの補助データとして有用 |
