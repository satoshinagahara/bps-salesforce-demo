# 製品改善ループ（Product Improvement Loop）設計ドキュメント

## 1. コンセプト

営業活動（顧客訪問・Web会議・電話等）で得られる顧客ニーズを構造化し、分析・意思決定・設計開発・営業フィードバックまでを一気通貫で回す仕組み。

```
 営業活動 → ニーズ蓄積 → 分析・可視化 → 施策起案 → 設計開発 → 営業へフィードバック
     ↑                                                              │
     └──────────────── 製品ロードマップとして還元 ←─────────────────────┘
```

### 背景と課題

- 営業が面談記録や議事録をSalesforceに残しても、顧客ニーズは**非構造化テキストに埋もれたまま**
- 同一顧客が繰り返し言及するニーズ、複数顧客に共通するニーズを発見するには、人が大量の活動記録を読む必要がある
- ニーズの全体像を定量的に把握し、ビジネスインパクトと紐づけて投資判断する手段がない
- 投資判断（施策起案）とその後の設計開発プロジェクトが分断され、顧客ニーズとの紐付けが失われる
- 営業側から「自社が何を開発しているか」が見えず、顧客提案に活かせない

### 解決アプローチ

**BI（ダッシュボード）が「What」を可視化し、AI（LLM）が「Why」を深掘りする。**

| 役割 | 担当 | 具体的な機能 |
|---|---|---|
| 全体傾向の可視化 | LWCダッシュボード | セグメント×製品×業種のクロス集計、鮮度スタックバー |
| 背景の要約・解釈 | Prompt Template (LLM) | セグメント内のニーズを読み込み、共通テーマ・顧客温度感・潜在機会を要約 |
| ニーズ抽出の自動化 | Prompt Template (LLM) | 面談記録から構造化ニーズを抽出、既存カードとの重複判定 |
| 施策提案 | Prompt Template (LLM) | 分析結果からタイトル・概要・根拠を生成 |
| 関連度評価 | Prompt Template (LLM) + SOSL | 施策に対する候補ニーズの関連度を多層評価 |

## 2. データ構造

### 2.1 オブジェクト一覧

| オブジェクト | API名 | 種別 | 役割 |
|---|---|---|---|
| 面談記録 | Meeting_Record__c | カスタム | 営業活動の記録（サマリー・文字起こし） |
| ニーズカード | Needs_Card__c | カスタム | 構造化された顧客ニーズ |
| ニーズカードソース | Needs_Card_Source__c | ジャンクション | ニーズカード↔面談記録の多対多関連 |
| 製品施策 | Product_Initiative__c | カスタム | 投資判断・製品戦略 |
| 施策ニーズ紐付け | Initiative_Need__c | ジャンクション | 施策↔ニーズカードの多対多関連 |
| 設計開発プロジェクト | Design_Project__c | カスタム | 施策を受けた設計開発の実行管理 |
| 設計フェーズ | Design_Phase__c | 子オブジェクト | プロジェクトのフェーズゲート管理 |

### 2.2 Meeting_Record__c（面談記録）

営業活動の記録を格納する。標準Task/Eventでは文字起こし等の大容量テキストに対応できないため、専用オブジェクトとして設計。

| フィールド | API名 | 型 | 説明 |
|---|---|---|---|
| レコード名 | Name | AutoNumber (MR-{0000}) | 自動採番 |
| 取引先 | Account__c | Lookup(Account) | 面談先顧客 |
| 商談 | Opportunity__c | Lookup(Opportunity) | 関連商談（任意） |
| 面談日 | Meeting_Date__c | Date | 面談実施日（必須） |
| 面談種別 | Meeting_Type__c | Picklist | 顧客訪問 / Web会議 / 電話 / 展示会 |
| 参加者 | Participants__c | LongTextArea(10,000) | 自社・顧客の参加者 |
| サマリー | Summary__c | LongTextArea(32,000) | 面談サマリー・議事録 |
| 文字起こし | Transcript__c | LongTextArea(131,072) | 大容量テキスト対応 |
| 抽出済み | Needs_Extracted__c | Checkbox | AI抽出の二重処理防止フラグ |

### 2.3 Needs_Card__c（ニーズカード）

面談記録からAIで抽出された構造化ニーズ。集合としてのシグナルに価値がある。

| フィールド | API名 | 型 | 説明 |
|---|---|---|---|
| レコード名 | Name | AutoNumber (NC-{0000}) | 自動採番 |
| タイトル | Title__c | Text(255) | ニーズの要約タイトル（必須） |
| 取引先 | Account__c | Lookup(Account) | 顧客 |
| ニーズ種別 | Need_Type__c | Picklist | 製品ニーズ / サービスニーズ / クレーム / 改善要望 / 新規案件 |
| 関連製品 | Product__c | Lookup(Product2) | 対象製品 |
| 詳細 | Description__c | LongTextArea(32,000) | ニーズの詳細説明 |
| 顧客の声 | Customer_Voice__c | LongTextArea(32,000) | 顧客の原文引用 |
| ビジネスインパクト | Business_Impact__c | Currency(18,0) | 金額規模（商談金額をコピー） |
| 優先度 | Priority__c | Picklist | 高 / 中 / 低（デフォルト: 中） |
| ステータス | Status__c | Picklist | 新規 / 確認済 / 対応中 / 対応完了 / 却下 |
| 抽出元面談 | Source_Meeting__c | Lookup(Meeting_Record__c) | 初回抽出元 |
| 統合元カード | Merged_From__c | Lookup(Needs_Card__c) | 自己参照（重複統合時） |
| 取引先レコードタイプ | Account_Record_Type__c | Text(100) | スナップショット（集計用） |
| 取引先種別 | Account_Type__c | Text(100) | スナップショット（集計用） |
| 取引先業種 | Account_Industry__c | Text(100) | スナップショット（集計用） |
| ビジネスユニット | Business_Unit__c | Text(100) | 商談のBUからコピー |

**設計判断:**
- ステータス管理は最小限。カードは大量に貯まるため個別レビューは非現実的。カードの価値は集合としてのシグナルにある。「却下」のみ残す
- ビジネスインパクトは商談金額をそのままコピー（按分不可のため）。merge時は大きい方で上書き
- Customer_Segment__c は廃止。Account側の3項目（RecordType/Type/Industry）をApexでスナップショットコピーに置換し、集計の柔軟性を確保

### 2.4 Needs_Card_Source__c（ニーズカードソース）

ニーズカードと面談記録の多対多関連を管理するジャンクション。1つのカードが複数面談で言及されるケースを追跡。

| フィールド | API名 | 型 | 説明 |
|---|---|---|---|
| レコード名 | Name | AutoNumber (NCS-{0000}) | 自動採番 |
| ニーズカード | Needs_Card__c | MasterDetail(Needs_Card__c) | 親カード |
| 面談記録 | Meeting_Record__c | Lookup(Meeting_Record__c) | ソース面談（必須） |

**用途:**
- 新規作成・merge両方でレコード追加
- 最新の Meeting_Record__r.Meeting_Date__c = 「最終言及日」として鮮度計算に使用
- 貢献者KPI（どの営業がどれだけニーズを拾ったか）の集計基盤

### 2.5 Product_Initiative__c（製品施策）

顧客ニーズに基づく投資判断・製品戦略を管理。

| フィールド | API名 | 型 | 説明 |
|---|---|---|---|
| レコード名 | Name | AutoNumber (PI-{0000}) | 自動採番 |
| 施策タイトル | Title__c | Text(255) | 施策名（必須） |
| 概要 (What) | What_Description__c | LongTextArea(10,000) | 何をするか |
| 根拠 (Why) | Why_Rationale__c | LongTextArea(10,000) | なぜやるか |
| 対象製品 | Product__c | Lookup(Product2) | 関連製品 |
| ステータス | Status__c | Picklist | 起案 / 評価中 / 承認 / 実行中 / 完了 / 不採用（必須） |
| 優先度 | Priority__c | Picklist | 高 / 中 / 低（デフォルト: 中） |
| ターゲット顧客 | Target_Customer__c | Text(255) | 想定顧客像 |
| ターゲット業種 | Target_Industry__c | Text(255) | 想定業種 |
| アプローチ (How) | Approach__c | LongTextArea(10,000) | 実現方法 |
| 着手目標日 | Target_Start__c | Date | 開発着手予定 |
| リリース目標日 | Target_Release__c | Date | リリース予定 |
| 見込みコスト | Estimated_Cost__c | Currency(18,0) | 投資額 |
| 見込み売上 | Estimated_Revenue__c | Currency(18,0) | 期待リターン |
| 判断理由 | Decision_Note__c | LongTextArea(10,000) | 承認/不採用の理由 |

**パスアシスタント（6ステージ）:**

| ステージ | ガイダンス概要 |
|---|---|
| 起案 | What/Whyを明確に記載、ニーズカード紐付けを確認、AI関連ニーズ候補から追加検討 |
| 評価中 | ターゲット具体化、投資対効果試算、アプローチ記載、設計開発プロジェクト有無確認 |
| 承認 | 判断理由記録、目標日設定、Design Project作成・紐付け、リソース確保 |
| 実行中 | Design Project進捗確認、スケジュール更新、ニーズ鮮度変化に注意 |
| 完了 | 成果記録、見込みとの差異振り返り、ニーズ充足確認 |
| 不採用 | 判断理由を必ず記録（将来の参考）。紐付けニーズは別施策で再検討可能 |

### 2.6 Initiative_Need__c（施策ニーズ紐付け）

施策とニーズカードの多対多関連を管理するジャンクション。

| フィールド | API名 | 型 | 説明 |
|---|---|---|---|
| レコード名 | Name | AutoNumber (IN-{0000}) | 自動採番 |
| 製品施策 | Initiative__c | MasterDetail(Product_Initiative__c) | 親施策 |
| ニーズカード | Needs_Card__c | Lookup(Needs_Card__c) | 紐付けニーズ |
| 関連理由 | Relevance_Note__c | Text(255) | 紐付け理由メモ |

### 2.7 Design_Project__c / Design_Phase__c（設計開発）

施策の「承認」後に作成される設計開発プロジェクトとそのフェーズゲート。

| フィールド (Design_Project__c) | API名 | 型 | 説明 |
|---|---|---|---|
| プロジェクト名 | Project_Name__c | Text(255) | 必須 |
| 製品施策 | Initiative__c | Lookup(Product_Initiative__c) | 起源となる施策 |
| 対象製品 | Product__c | Lookup(Product2) | 開発対象 |
| 設計BOM | BOM_Header__c | Lookup(BOM_Header__c) | E-BOM |
| ステータス | Status__c | Picklist | 企画中 / 進行中 / 完了 / 中止 |
| 計画開始/終了日 | Planned_Start__c / Planned_End__c | Date | 計画期間 |
| 実開始/終了日 | Actual_Start__c / Actual_End__c | Date | 実績期間 |

| フィールド (Design_Phase__c) | API名 | 型 | 説明 |
|---|---|---|---|
| 設計開発プロジェクト | Design_Project__c | MasterDetail | 親プロジェクト |
| 表示順 | Phase_Order__c | Number(3,0) | フェーズ順序（必須） |
| ステータス | Status__c | Picklist | 未着手 / 進行中 / 完了 / 遅延 |
| ゲートレビュー結果 | Gate_Result__c | Picklist | 合格 / 条件付合格 / 差戻し / 未実施 |

                           │（設計フェーズ）         │

```

**リレーションのポイント:**
- Meeting_Record__c → Needs_Card__c: AIが面談テキストからニーズを抽出して生成
- Needs_Card_Source__c: 同じニーズが複数面談で言及された場合に追加（鮮度追跡の基盤）
- Initiative_Need__c: 施策がどの顧客ニーズに根拠を持つかを可視化（投資判断の説明責任）
- Design_Project__c → Product_Initiative__c: 施策の実行フェーズとして紐付け
- Product2: ニーズカード・施策・設計プロジェクトを横断する共通軸

## 4. 画面構成

### 4.1 ニーズ分析ダッシュボード（needsAnalysisDashboard）

**配置:** ホームページ / アプリケーションページ
**対象ユーザー:** 製品企画、営業企画、マネジメント

セグメント横断でニーズカードを可視化・分析する中核画面。

```
┌──────────────────────────────────────────────────────┐
│ ニーズ分析ダッシュボード                               │
├────────────┬────────────┬────────────────────────────┤
│ [顧客]     │ [サプライヤー] │ [全体]        ← セグメント切替│
├────────────┴────────────┴────────────────────────────┤
│ 期間: [全期間] [3M] [6M] [1Y]    指標: [件数] [金額]  │
├──────────┬──────────┬──────────┬─────────────────────┤
│  ニーズ   │ 優先度:高 │ インパクト │ 鮮度 (Fresh/Aging/ │

├──────────┴──────────┴──────────┴─────────────────────┤

└──────────────────────────────────────────────────────┘

**機能:**
- **セグメント切替:** Account_Record_Type__c を動的収集し、顧客/サプライヤー/全体などのタブを自動生成
- **期間フィルタ:** 全期間 / 3ヶ月 / 6ヶ月 / 1年
- **指標切替:** 件数 / ビジネスインパクト金額
- **KPIサマリー:** 総件数、高優先度件数、金額合計、鮮度分布
- **5つのビュー:**
  1. 製品Family別バーチャート（鮮度スタックバー）
  2. 製品×ニーズ種別クロス集計（ヒートマップ）
  3. 製品×業種クロス集計
  4. 業種×ニーズ種別クロス集計
  5. 顧客Top10バーチャート
- **AI分析:** バー/セルクリックで該当セグメントのニーズをLLMが分析、インサイトをインライン表示
- **施策起案:** AI分析結果から「施策を起案」ボタンでモーダルを開き、AIが施策提案を自動生成

### 4.2 関連ニーズ候補（initiativeNeedsMatcher）

**配置:** Product_Initiative__c レコードページ
**対象ユーザー:** 製品企画

施策に関連するニーズカードを検索・評価・紐付けする画面。

```
┌──────────────────────────────────────────────────────┐
│ 関連ニーズ候補                          紐付け済み: 3件 │
├──────────────────────────────────────────────────────┤
│ 候補 12件   [全選択]      [AI関連度分析] [選択した3件を紐付け]│
├──────────────────────────────────────────────────────┤
│ ▼ AI関連度分析                                       │
│  ## 関連度サマリー                                    │
│  本施策はソーラーパネルの高効率化に関連し...              │
│  ## 高関連                                           │
│  ・NC-0012: 変換効率95%以上の要求...                   │
│  ## カバレッジ評価                                    │
│  候補の60%が直接関連...                                │
├──────────────────────────────────────────────────────┤
│ ☑ NC-0012  [関連度: 高] [優先度: 高]  製品ニーズ       │
│   変換効率95%以上のソーラーパネル要求                    │
│   ノヴァテック │ ソーラーパネル │ ¥2.4億               │
│                                                      │
│ ☐ NC-0015  [関連度: 中] [優先度: 中]  改善要望         │
│   パネル軽量化による施工コスト削減                       │
│   丸菱商事 │ ソーラーパネル │ ¥8,000万               │
└──────────────────────────────────────────────────────┘
```

**機能:**
- **多層検索:** Tier 1 = 同一製品（Product__c一致 → 関連度「高」）、Tier 2 = SOSLキーワード検索（施策タイトル・製品名・概要から抽出 → 関連度「中」）
- **バッジ表示:** 関連度バッジ（高=緑/中=青）と優先度バッジ（高=赤/中=黄/低=緑）を独立表示
- **キーワード抽出:** 日本語テキストを記号・句読点で分割、ストップワード除去、上位10ワードに制限
- **AI関連度分析:** ボタンクリックで全候補の関連度をLLMが評価（高/中/低 + カバレッジ%）
- **一括紐付け:** チェックボックスで選択した候補をInitiative_Need__cとして一括作成

### 4.3 製品ロードマップ（opportunityRoadmap）

**配置:** Opportunity レコードページ
**対象ユーザー:** 営業

商談に関連する製品施策と設計開発プロジェクトを一覧表示。営業が顧客に「自社の開発計画」を説明する際の情報源。

**機能:**
- 商談の製品（OpportunityLineItem）から Product Family を特定
- 同じ Product Family の Product_Initiative__c と Design_Project__c を検索・表示
- 施策のステータス・優先度・目標リリース日を一覧表示

## 5. AI活用ユースケース

### 5.1 ニーズ抽出（NeedsCardExtraction）

| 項目 | 内容 |
|---|---|
| トリガー | 面談記録ページの「ニーズ抽出」ボタン（Screen Flow）/ バッチ処理ランチャー（NeedsCardBatch） |
| 入力 | 面談サマリー + 文字起こし + 顧客情報 + 既存ニーズカード一覧 |
| モデル | GPT-4o（高精度が必要なため） |
| 処理 | 面談内容から各ニーズを識別し、既存カードとの意味的重複判定を含めて一括判断 |
| 出力 | JSON: 各ニーズについて新規作成 / 既存カードに統合 / スキップの3択 |
| 後続処理 | Apex が Needs_Card__c の insert/update + Needs_Card_Source__c 作成 |

```
[面談記録ページ] → 「ニーズ抽出」ボタン押下
  1. Apex: Meeting_Record__c の Summary/Transcript + Account + Opportunity 取得
  2. Apex: 同一顧客の既存 Needs_Card__c を取得 (Status <> '却下')
  3. Prompt Template: 面談内容 + 既存カード → LLM が一括判断
     - 新しいニーズ → 新規作成
     - 既出ニーズ → 既存カードに情報統合（言及頻度が優先度の裏付けに）
     - 抽出不要 → スキップ
  4. Apex: DML実行 + Needs_Extracted__c = true
  5. 結果表示: 「○件新規作成、○件統合、○件スキップ」
```

**重複判定の設計:**
- **顧客内:** 重複排除。同一顧客の同じニーズは統合（SOQLで既存取得 + LLMで意味的判定）
- **顧客間:** 独立して存在。同一ニーズを持つ顧客数 = 市場トレンドの指標
- **SOSLではなくLLM判定:** 顧客あたり数十件程度のため `WHERE Account__c = :accountId` で十分。意味的重複（「電源効率を上げてほしい」≒「消費電力を下げたい」）はキーワードでは捉えられない

### 5.2 セグメント分析（NeedsAnalysisInsight）

| 項目 | 内容 |
|---|---|
| トリガー | ダッシュボードのバー/セルクリック |
| 入力 | 該当セグメントのニーズカード群（タイトル・詳細・顧客の声・金額・鮮度情報） + 関連施策の状況 |
| モデル | GPT-4o mini（コスト効率重視） |
| 出力 | 共通テーマ / 注目シグナル / 顧客温度感 / 鮮度シグナル / 関連施策状況 / 潜在機会 / リスク |
| 設計方針 | 事実のみ出力（提案・推薦は含めない）、400文字以内 |

### 5.3 施策起案サジェスト（InitiativeSuggestion）

| 項目 | 内容 |
|---|---|
| トリガー | AI分析パネルの「施策を起案」ボタン |
| 入力 | セグメントのニーズカード群（分析コンテキストと同等） |
| モデル | GPT-4o mini |
| 出力 | 【タイトル】（30文字以内）/ 【概要】（200文字、What）/ 【根拠】（200文字、Why） |
| 設計方針 | データドリブン（件数・顧客数・金額・鮮度を根拠に使用）、推測を含めない |

### 5.4 施策ニーズ関連度分析（InitiativeNeedsRelevance）

| 項目 | 内容 |
|---|---|
| トリガー | 関連ニーズ候補パネルの「AI関連度分析」ボタン |
| 入力 | 施策情報（タイトル・概要・根拠・製品）+ 候補ニーズカード群 |
| モデル | GPT-4o mini |
| 出力 | 関連度サマリー / 高関連（カードID・理由）/ 中関連 / 低関連 / カバレッジ評価 |
| 設計方針 | カードIDとタイトルを引用必須、500文字以内 |

### 5.5 鮮度（Freshness）の仕組み

ニーズカードの「鮮度」は、そのニーズが顧客から最後に言及された時期を示す。

```
鮮度 = Needs_Card_Source__c → Meeting_Record__r.Meeting_Date__c の最新日
```

| 区分 | 条件 | 表示色 | 意味 |
|---|---|---|---|
| Fresh | 最終言及日が3ヶ月以内 | 緑 | 顧客が現在も関心を持っている |
| Aging | 最終言及日が3〜6ヶ月前 | 黄 | 関心が薄れている可能性 |
| Stale | 最終言及日が6ヶ月超 | 灰 | 状況が変化している可能性が高い |

ダッシュボードのバーチャートは鮮度スタックバーとして表示され、AI分析にも鮮度情報がコンテキストとして渡される。

## 6. 想定利用ユーザーと運用フロー

### 6.1 ユーザーロール別の利用

| ロール | 主な利用画面 | 主な操作 |
|---|---|---|
| **営業担当** | 面談記録ページ / Opportunityページ | 面談記録入力 → ニーズ抽出ボタン / ロードマップ確認 |
| **営業マネージャー** | ダッシュボード | セグメント分析、顧客別ニーズ傾向把握 |
| **製品企画** | ダッシュボード / 施策レコードページ | ニーズ分析 → 施策起案 → ニーズ紐付け → 承認プロセス |
| **開発マネージャー** | 施策レコードページ / Design Projectページ | 承認済み施策の設計プロジェクト作成・進捗管理 |
| **経営層** | ダッシュボード | 全体傾向の把握、投資判断の裏付け確認 |

### 6.2 運用フロー

```
【Phase 1: インプット（日常業務）】
  営業担当が顧客と面談
    → Meeting_Record__c にサマリー・議事録を入力
    → 「ニーズ抽出」ボタンでAIが構造化カードを自動生成
    → 営業担当は結果を確認するだけ（追加作業不要）

【Phase 2: 分析（週次〜月次）】
  製品企画がダッシュボードで全体傾向を確認
    → セグメント/製品/業種の軸でニーズ分布を把握
    → 気になるセグメントをクリックしてAI分析
    → 共通テーマ・顧客温度感・潜在機会を把握
    → 鮮度から「今ホットな」ニーズと「陳腐化した」ニーズを識別

【Phase 3: 意思決定（四半期〜随時）】
  分析結果から施策を起案
    → ダッシュボードの「施策を起案」でAIが提案を自動生成
    → または手動で Product_Initiative__c を作成
    → 関連ニーズ候補パネルでAIが関連度を評価
    → ニーズカードを紐付けて投資判断の根拠を明示
    → パスアシスタントに沿って 起案→評価中→承認 を進行

【Phase 4: 実行（承認後）】
  承認された施策に Design_Project__c を紐付け
    → 設計フェーズのゲートレビューで進捗管理
    → 施策ステータスを「実行中」に更新

【Phase 5: フィードバック（営業への還元）】
  営業が Opportunity ページの製品ロードマップを確認
    → 「この顧客のニーズに対応する開発が進行中」を把握
    → 顧客への提案活動に活かす
    → 新たな面談で得たフィードバック → Phase 1 に戻る
```

## 8. 技術的な設計判断

### ConnectApi パターン
Agentforceのprimitive://String入力には255文字制限があるため、Apex内で `ConnectApi.EinsteinLLM.generateMessagesForPromptTemplate()` を直接呼び出す統合パターンを採用。大量のニーズカードデータをPrompt Templateに渡す際の制約を回避。

### SOSLキーワード検索（多層マッチング）
InitiativeNeedsMatcherControllerでは、単純なProduct Family一致では広すぎるため、多層検索を採用:
- **Tier 1（高関連）:** 同一Product__c の完全一致
- **Tier 2（中関連）:** 施策のTitle/Product名/What_Descriptionから日本語キーワードを抽出し、SOSLで全文検索

キーワード抽出は記号・句読点で分割、2文字以上の単語を採用、ストップワード除去、上位10ワードに制限。

### 鮮度計算
Needs_Card_Source__c → Meeting_Record__r.Meeting_Date__c のサブクエリで最新言及日を取得。現規模（数十〜数百件）ではApexループ集計で問題ないが、万単位になった場合はSOQL GROUP BYへの移行を想定。

### Prompt Template のActivate
Metadata APIでデプロイしても `IsActive=false` のまま。Prompt Builder UIで手動Activateが必要（ConnectApiから呼ぶとgenerationsが空リストになる）。

## 9. 将来構想: RAGによるセマンティック関連度マッチング

### 現状の課題
現在の施策↔ニーズカードのマッチングは、Tier 1（同一Product__c）+ Tier 2（SOSLキーワード検索）の2層構造。SOSLはあくまで字句一致ベースのため、表現が異なるが意味的に関連するニーズ（例:「電源効率を上げてほしい」≒「消費電力を下げたい」）を捉えきれない。

### RAG活用案
Salesforce Data Cloud（旧CDP）のベクトルデータベース機能、またはEinstein Search Indexを活用し、ニーズカードのエンベディング（ベクトル化）を事前に行う。

```
【インデックス構築（バッチ/トリガー）】
  Needs_Card__c の Title + Description + Customer_Voice を結合
    → Embedding API でベクトル化
    → Data Cloud Vector Store に格納

【検索時（施策レコードページ）】
  Product_Initiative__c の Title + What_Description + Why_Rationale を結合
    → Embedding API でクエリベクトル生成
    → Vector Store に対してコサイン類似度検索（Retriever）
    → スコア付きで上位N件を返却

【マッチング精度の多層化】
  Tier 1: 同一Product__c → 関連度「高」（現行維持）
  Tier 2: RAGセマンティック検索 → コサイン類似度スコアで「高/中/低」を判定
  Tier 3: SOSLキーワード検索 → フォールバック（RAG未ヒット分の補完）
```

### 期待される効果
- **意味的マッチング:** 「高効率化」と「省エネ」、「コスト削減」と「価格競争力」など、表現が異なるニーズを関連付けられる
- **スコアによる定量的な関連度:** 現在のTier分類（高/中の2段階）から、連続値スコアによるランキングが可能に
- **施策起案時の網羅性向上:** キーワードに現れない潜在的な関連ニーズも候補に浮上する

### Salesforce上の実現方式（候補）
1. **Data Cloud + Einstein Embedding:** Data CloudにNeeds_Card__cをストリーミングし、Vector Indexを構築。Apex ConnectApiからRetriever呼び出し
2. **Einstein Search（将来）:** Salesforce標準のセマンティック検索が拡張されれば、SOSL相当の使い勝手でベクトル検索が可能になる見込み
3. **外部Vector DB連携:** Pinecone等の外部サービスをNamed Credential経由で呼び出す方式（Data Cloud未契約の場合の代替）

### 実装上の考慮点
- エンベディングの更新タイミング: カード作成/更新時のトリガーまたは定期バッチ
- 日本語エンベディングの品質: 多言語対応モデル（multilingual-e5等）の選定が重要
- Data Cloudのライセンス要件とAPIコールアウト制限の確認が必要
